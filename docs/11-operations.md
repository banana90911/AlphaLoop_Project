## 운영 (1인 무인 운영의 *실패 차단* 단일 소스)

> 매매 로직이 옳아도 *운영*이 무너지면 사고난다. 본 문서는 "시스템을 안 죽고, 두 번 주문 안 가게, 조용히 실패하지 않게 돌리는 법"의 단일 소스다 — *전략이 진짜 돈을 버는지*의 검증 게이트와는 별개 관심사다. 세 장으로 구성된다 — **11-1 비용 통제**, **11-2 운영 신뢰성 불변식**, **11-3 LLM 호출 운영**.
>
> 본 문서는 *무엇을·어떻게 보장하는가*(원칙·정책)만 규정한다. 그것을 *어느 파일·함수가 담당하는가*는 규정하지 않으며, 모듈 구성은 리포 구조를 확정하는 착수 단계에서 정한다.

## 11-2. 운영 신뢰성 불변식 (1인 무인 운영의 *실패 차단*)

> 매매 로직이 옳아도 운영이 무너지면 사고난다. 1인 무인 운영의 실패는 거의 항상 *전략의 문제가 아니라 운영의 문제*(silent failure·키 만료·디스크 풀·시계 어긋남·부분 체결 reconcile)에서 온다. 본 장은 그 전형적 실패 시나리오 각각에 대해 *코드·설정·체크리스트가 무엇을 어떻게 박아두는가*를 결정 단일 소스로 명시한다. 본 장의 항목은 *원칙*이고, 구체 라이브러리 선택은 기술스택 장에, 미확정 결정은 *열린 항목*(추후 결정) 문서에 둔다.

### 11-2.1 사이클 idempotency — 사이클 도중 죽어도 두 번 주문 안 가게

**문제 시나리오**: 9단계 중 8단계(주문 실행)는 성공해 KIS 체결까지 됐는데 9단계(기록) 직전 프로세스가 죽으면 어떻게 되는가? SQLite에는 주문 정보가 없는데 KIS 잔고에는 체결이 있다. 다음 사이클이 시작되면 잔고 정합성 게이트(7단계 선행)가 *불일치*를 잡아 SafeStop을 띄운다. 운영자가 수동 reconcile하기 전엔 사이클이 안 돈다 — 그 사이에 시장은 계속 움직이고 손절·진입 기회가 사라진다. 이걸 결정론으로 풀어야 한다.

**원칙 — 정확히 한 번(exactly-once) 주문 송출의 *효과* 보장**:

- 사이클 시작 시 *cycle_id*(timestamp 기반) 발급 → SQLite `cycles` 테이블에 *intent* 상태로 적재(주문 시도 *전*). 이로써 *우리가 무엇을 하려 했는지*는 디스크에 남는다.
- `cycles.status`는 `intent` → `ordering`(주문 송출 시작) → `recorded`(9단계 기록 완료) → `failed`(중단) 의 4단계 상태머신
- 각 주문에 *client_order_id*(예: `{cycle_id}-{symbol}-{side}-{seq}`) 부여. KIS API가 직접 지원하면 그대로 사용, 안 하면 *자체 메타*로 SQLite에 저장해 사후 매칭에 쓴다
- **사이클 재진입 복구**: 사이클 시작 시 *복구 루틴*이 `intent`/`ordering` 상태로 남은 직전 cycle을 찾아, 그 cycle의 *주문 의도와 KIS 실체결을 비교*해 재동기화한 뒤 정상 사이클을 시작한다. KIS 측 사후 조회 경로는 *주식일별주문체결조회* API(모의 지원, `VTTC0081R`/`TTTC0081R`)로 직전 시각 이후 발생한 우리 계좌 체결을 받아 자체 cycle_id와 매칭

**구현**: 사이클 진입점이 이 *복구 루틴*을 *가장 먼저* 호출한다. 리스크 엔진의 선행 게이트(잔고 정합성)는 이 복구 이후에 동작하므로 정상 복구 후에는 SafeStop이 뜨지 않는다.

### 11-2.2 SQLite 동시성 — WAL 모드 강제

**문제 시나리오**: 매매 코어(write), 회고 에이전트(write), 대시보드(read)가 동시에 같은 SQLite 파일을 잡는다. 기본 모드(rollback journal)는 *writer 1명이 잡으면 reader 전부 차단*이라, 대시보드 새로고침이 사이클 commit과 겹치면 `database is locked` 에러가 나거나 9단계 기록 일부가 깨질 수 있다.

**원칙 — 모든 연결 초기화 시 *강제* PRAGMA**:

- `PRAGMA journal_mode=WAL` — Write-Ahead Logging 모드. writer 1명 + reader 다수가 *동시 진행 가능*. 우리 시스템의 정확한 동시성 패턴(매매 코어 write + 대시보드 read)에 최적
- `PRAGMA busy_timeout=5000`(5초) — 일시 잠금 시 즉시 에러 내지 않고 5초간 자동 재시도
- `PRAGMA synchronous=NORMAL` — WAL 모드에 적절한 안전성·성능 절충(FULL은 과보호, OFF는 위험)
- `PRAGMA foreign_keys=ON` — `decisions ↔ agent_predictions` 등 관계 무결성 강제
- WAL 파일 checkpoint는 자동(`PRAGMA wal_autocheckpoint=1000`)으로 두고, *주간 1회 명시적* `PRAGMA wal_checkpoint(TRUNCATE)`로 WAL 파일 크기 폭주 방지
- **Single-writer 정책**: 매매 코어·회고 에이전트는 write 허용. 대시보드는 *읽기 전용* 연결(`?mode=ro` 또는 `PRAGMA query_only=1`) — 대시보드 버그가 학습 데이터를 망가뜨릴 수 없게 격리

**구현**: *단일 연결 헬퍼*가 모든 PRAGMA를 일괄 적용한다. PRAGMA가 누락된 연결이 생기지 않도록 *직접 `sqlite3.connect()` 호출은 금지*하고 이 헬퍼(또는 컨텍스트 매니저)만 쓰며, `ruff` 룰로 기계적으로 차단한다.

### 11-2.3 KIS 운영 정책 — 토큰·rate limit·WebSocket·5xx

KIS API의 운영 현실은 다음과 같다(KIS 명세서·실증 기준): 접근토큰 24시간 만료, REST 호출 한도(실전 초당 약 20건, 모의는 더 낮음 — 정확 수치는 외부 API 레퍼런스 문서 참조), WebSocket 세션당 41종목 한도, 모의계좌 분기 단위 갱신, 일시적 5xx 점검창. 이를 *결정론 정책*으로 흡수한다.

- **토큰 자동 갱신**: *KIS 클라이언트*가 만료 5분 전에 자동 재발급. 발급 실패 시 *지수 백오프*(1초→2초→4초) 3회 재시도, 모두 실패하면 SafeStop. 토큰 발급 직후 다음 만료까지의 잔여시간을 SQLite에 적재해 사후 추적 가능
- **Rate limiter**: 모든 KIS 호출 전 Token Bucket(`aiolimiter` 또는 `asyncio.Semaphore`)을 통과시켜 *초당 한도의 80%까지*만 사용(안전 마진). 한도는 *외부 설정*으로 분리하고 *모드별 한도가 다르므로* 자동 설정
- **WebSocket 자동 재접속**: 끊김 감지 시 *exponential backoff* 재접속 + 끊김 동안 발생한 시세는 사이클 시작 시 REST polling으로 보완 갱신. 재접속 후 *구독 종목 재등록*은 자동
- **모의계좌 만료 알림**: 모의 토큰 발급 시 *만료일 컬럼*을 SQLite에 적재. *D-30·D-7·D-1* 시점에 대시보드와 운영 알림 채널로 경고 송출
- **5xx 응답 처리**: 일시적(503·504)은 *지수 백오프 3회* 재시도. 영구적(404·401·403)은 즉시 SafeStop. **주문 송출 호출(POST `/order-cash` 등)은 *재시도 금지*** — 중복 주문 위험이 idempotency 키로 검증되지 않는 한 손실보다 크다. **단 송출 실패는 *접수 여부 불확실* 상태이므로, 재시도(보내기) 대신 *그 사이클 내에서 즉시 1회 주식일별주문체결조회(`VTTC0081R`/`TTTC0081R`, 모의 지원)로 접수 여부를 확인*한다(조회는 중복주문 위험이 없어 안전)**. 접수됐으면 `trades`·`positions`를 동기화하고 *그 매수에 대한 손절 스톱 등록 여부까지 확인*(미등록이면 즉시 등록 — 5-1 A "맨몸 포지션 방지"), 미접수면 정상 무거래 처리. 이로써 *다음 정기 사이클까지(최대 4.5시간) 손절 없는 맨몸 포지션이 방치되는 갭*을 사이클 내에서 닫는다. 즉시 조회로도 불확실하면(조회 자체가 5xx) *사이클 재진입 복구 절차*로 사후 reconcile

### 11-2.4 Heartbeat·dead-man's switch — silent failure 차단

**문제 시나리오**: cron이 안 돌았거나 토큰 발급 실패로 사이클이 30초 만에 죽어도, 우리는 SQLite에 *아무 기록이 없는* 상태가 된다. 정기 사이클이라 *시각이 지나갔는데 기록이 없다*는 사실 자체가 신호인데, 그 신호를 *우리가 보는* 게 핵심이다. 외부 모니터링이 없으면 며칠 뒤 대시보드를 열어보고서야 알게 된다.

**원칙 — 외부 모니터링에 "살아 있다" 신호 보내기 (dead-man's switch)**:

- 사이클 *정상 종료* 시점에 외부 모니터링(healthchecks.io·Better Stack·UptimeRobot·자체 cron-ping 중 하나는 *열린 항목*에서 확정)으로 HTTPS ping 송출
- 모니터링 측이 *예상 시각 + 그레이스 윈도우(예: 정기 사이클 +15분)* 내 ping 미수신 시 *자동 알림*. ping이 오면 정상, 안 오면 알람
- **알림 채널 — Discord**: 모든 운영 알림을 Discord 웹훅으로 보낸다. Discord 앱은 폰·데스크톱·웹에서 같은 알림을 받으므로, 단일 채널이라도 휴대폰 분실·단일 기기 장애 시 다른 기기로 확인 가능
- *정상 종료 ping*과 *SafeStop ping*은 다른 endpoint를 써 구분. SafeStop은 즉시 알람, 정상은 미수신 시에만 알람
- 조기 경보 신호도 같은 채널로: 모의계좌 만료 임박, KIS rate limit 90% 도달, SQLite 디스크 80% 사용, 백업 실패, NTP 동기화 오차

### 11-2.5 시크릿 관리 — 단일 파일 평문 금지

**문제 시나리오**: KIS App Key/Secret, FRED API 키, DART 키, healthchecks.io 토큰 등이 노트북·서버에 평문으로 떨어지면 *도난 1회로 자본 전체가 위험*에 노출된다. 또한 실수로 git push·스크린샷 공유 같은 사고로도 노출된다.

**원칙**:

- **첫 배포 환경별 시크릿 저장소** (Phase 0 확정): **NCP 서버 = 서버 로컬 `.env`(`chmod 600` + 서버 디스크 암호화)**. 더 먼 목표 맥미니=Keychain. 클라우드 Secret Manager(AWS/GCP/NCP)는 우리 규모엔 과잉이라 *권한 제한된 서버 `.env`로 충분*하다.
- `.env` 파일은 *로컬 개발*과 *NCP 서버 운영* 모두에 쓰되, `.gitignore`로 리포 유출을 강제 차단하고 서버에서는 권한을 제한(`chmod 600`)한다. 공개 러너(GitHub Actions 등) 일회성 환경은 쓰지 않으므로 '러너에 평문 `.env` 금지'식 제약은 적용 대상이 없다 — 단 *로그·백업에 시크릿 평문 노출 금지* 원칙은 그대로 유지(마스킹·클라이언트 측 암호화)
- `pydantic-settings`로 환경변수에서만 로딩. *시크릿이 SQLite에 적재되는 경로 차단* — 계좌번호도 마스킹해 저장(예: `810XXXXX` 형태)
- **로그 마스킹**: structlog 로그에서도 App Key/Secret·계좌번호·잔고는 마스킹. structlog processor로 자동 처리
- **시크릿 회전 정책**: KIS App Key/Secret은 6개월마다 자체 회전, FRED·DART·healthchecks는 노출 의심 시 즉시 회전. 회전 일정을 분기 운영 점검 체크리스트에 포함

### 11-2.6 백업·암호화·재해 복구

**문제 시나리오**: 5년치 SQLite 파일(자본곡선·체결·교훈 일체)이 단일 디스크에 의존한다. 디스크 사망·랜섬웨어·도난·실수 삭제 시 학습 자산 영구 손실. 이는 학습 루프의 *5년차 보정 통계 도달*이라는 약속을 통째로 깬다.

**원칙 — 3-2-1 백업 + 암호화 + 정기 복구 리허설**:

- **온라인 백업**: 매 사이클 종료 후 `sqlite3 .backup` API(또는 `litestream`으로 연속 백업)로 atomic 스냅샷 생성. 백업 중에도 매매·대시보드 동작 가능
- **3-2-1 백업**: 원본(맥미니/서버) + 로컬 2차(외장 SSD·NAS) + 클라우드(S3·Backblaze·iDrive 등, *클라이언트 측 암호화* 후 업로드). 세 사본이 각각 다른 장애에 독립적이어야 함
- **암호화** (Phase 0 결정, *열린 항목*): SQLCipher(DB 자체 암호화) 또는 OS 디스크 암호화(macOS FileVault·LUKS) 중 한 가지. *추천*: macOS FileVault가 우리 규모에선 충분하고 운영 부담이 작음
- **복구 리허설**: 분기마다 백업에서 *복원해 사이클 1회 정상 동작 확인* 체크리스트 실행. RPO(허용 가능 데이터 손실 시간) 1일, RTO(허용 가능 복구 시간) 4시간 목표. 리허설 결과를 `safe_stop_events` 테이블과 별도의 운영 일지에 기록
- **법적·세무 데이터 export**: 매년 1회 `trades`·`outcomes` 전체를 CSV로 export해 양도세·금융투자소득세 신고 자료로 별도 보관. 신고용 export는 *원본 SQLite와 독립적*으로 보관해 데이터 손실 시에도 신고 자료가 살아남음

### 11-2.7 시간대 정규화 경계

**문제 시나리오**: KRX=KST(UTC+9), FRED·yfinance=UTC/ET, KIS 응답 형식이 시리즈마다 다르다. naive datetime(시간대 정보 없는 datetime)이 한 번이라도 섞이면 사이클 스킵 로직·이벤트 캘린더가 잘못 분기한다 — 예컨대 KST 10:00을 UTC 10:00으로 오인하면 9시간 차이로 *사이클이 안 도는 시각에 돌거나 도는 시각에 안 돈다*.

**원칙 — 내부 표준은 UTC + tz-aware, 경계 함수만이 KST 변환 책임**:

- *내부 표준*: 코드 안의 모든 timestamp는 *UTC + tz-aware*(`datetime.datetime.now(datetime.UTC)`). 사용자에게 보여줄 때만 KST로 변환
- *경계 함수* 한 쌍만이 *KIS↔내부* 시간대 변환을 담당한다. 다른 코드는 이 경계 함수만 호출하고 직접 시간대 변환을 하지 않는다
- yfinance·FRED 응답은 라이브러리가 이미 tz-aware 반환 — 그대로 사용. *naive로 들어오는 경우는 명시적으로 거부*(에러)
- 거래일 판정은 `exchange_calendars` 라이브러리에 위임. *KRX 휴장 + 임시휴장*은 KIS 영업일 API와 2중 점검 (라이브러리 사후 반영 누락 대비)
- **CI 룰**: `ruff` 또는 `mypy` 플러그인으로 `datetime.datetime.now()`(인자 없음, naive) 호출을 *코드 전체에서 금지*. `datetime.datetime.now(UTC)`만 허용

### 11-2.8 NTP·시계 동기화

**문제 시나리오**: 손절 트리거·세션 라벨(morning/afternoon)·이벤트 윈도우·heartbeat 모두 *시계*에 의존한다. 시계가 1분 어긋나면 종가 컷오프 오판으로 잘못된 데이터로 결정, 세션 라벨 잘못 적재로 보정 통계 차원 오염.

**원칙**:

- 시스템 NTP 동기화 강제. macOS는 `시스템 설정 → 일반 → 날짜 및 시간 → 자동으로 설정` ON. Linux는 `timedatectl set-ntp true`
- 사이클 시작 시 *NTP 동기화 상태 점검*(`ntpdate -q` 또는 OS API). 오차 ±5초 초과 시 *경고 로그* 송출 + 그 사이클은 *동기화 후 재시작*
- 클라우드(AWS·GCP)는 자체 NTP 제공 — 동기화는 기본이지만 *점검 코드는 동일하게 유지*(환경 무관 동작 원칙)

### 11-2.9 "if 실전 then" 분기 금지의 강제 메커니즘

**문제 시나리오**: 설계 불변식("모의/실전은 config 플래그 하나로 전환, `if mode == 'live':` 같은 분기 없음")을 *원칙*만 적어두면 1년 후 누군가(=본인)가 슬쩍 모드 분기를 박는다. 한 번 박히면 모의에서 검증한 코드와 실전에서 도는 코드가 *조용히 갈라지기* 시작한다.

**원칙 — CI 룰로 기계적으로 차단**:

- `ruff` 룰 또는 grep 기반 CI 체크: `if .*(live|paper|backtest).*:` 패턴이 코드(테스트·문서·설정 제외)에 등장하면 *PR 차단*
- 예외 허용 경로 화이트리스트: *설정·브로커 클라이언트(엔드포인트 주입)·문서·테스트* 경로만 패턴 허용. 다른 곳은 일괄 차단
- **통합 테스트 매트릭스**: 모든 핵심 모듈(파이프라인·리스크 엔진·청산·인출 등)을 *paper·live 두 모드*에서 *같은 골든 케이스*로 실행, 결과 동일성 검증. 두 모드 결과가 *바이트 단위로 같지 않으면* 분기가 숨어 있다는 신호
- 모드 차이는 *config 값*(KIS 도메인·계좌번호·rate limit 한도 등)으로만 흡수 — 코드는 모드를 모른다

### 11-2.10 로그 회전·보존

**문제 시나리오**: structlog JSON 로그가 무한 누적되면 디스크 풀로 사이클이 죽는다.

- 일별 회전(`logging.handlers.TimedRotatingFileHandler` 또는 `loguru`)
- 보존: 최근 30일 raw, 31~365일 압축(gzip), 365일 이후 삭제. 학습용 1차 자료는 SQLite에 별도 적재돼 있으므로 로그는 디버깅·운영용으로만 본다
- 디스크 80% 사용 시 heartbeat 채널로 경보

(SQLite 연결 통일·직접 `sqlite3.connect()` 금지는 *WAL 동시성* 항목에 둔다.)

### 11-2.11 DB 마이그레이션

**문제 시나리오**: 학습 데이터 모델은 시간이 지나면 *컬럼 추가·계층 압축 변화·`source` 등 메타 컬럼 추가*가 잦다. 스키마 변경을 손코딩하면 5년치 학습 데이터가 깨질 위험.

**원칙**:

- *Alembic*(SQLAlchemy의 마이그레이션 도구지만 ORM 없이도 단독 사용 가능) 또는 *순번 SQL 파일* 방식의 자체 마이그레이션 스크립트
- 각 마이그레이션은 *forward + backward* 둘 다 작성(롤백 가능). 운영 중 마이그레이션이 실패하면 *직전 상태로 복귀* 가능해야 함
- 모든 학습 테이블에 `schema_version` 컬럼(기본 1), 마이그레이션 시 증가
- *마이그레이션 실행 전 자동 백업* 생성(백업·복구와 동일 메커니즘)
- 마이그레이션 실행 후 *데이터 정합성 체크 SQL* 실행 (예: 누락 컬럼이 모두 채워졌는가)

### 11-2.13 무중단 배포 정책

**문제 시나리오**: 코드 변경 시 매매 사이클이 *진행 중*일 수도 있다. 새 코드가 진행 중인 cycle을 망가뜨리면 *사이클 복구 절차*로도 못 살릴 수 있다.

**원칙**:

- **배포 윈도우**: 평일 11:00~13:00(정기 사이클 사이) 또는 16:00 이후(장 마감 후). 사이클이 *현재 도는 시각*에는 배포 금지
- 배포 시 *진행 중 cycle이 있는지 확인* — SQLite `cycles` 테이블의 `intent`/`ordering` 상태 행이 있으면 *완료 대기* 또는 *SafeStop 후 배포*
- *DB 마이그레이션*이 있는 배포는 *마이그레이션 자동 백업·정합성 체크*가 통과한 후에만 진행
- **롤백 절차**: 직전 git tag로 즉시 복귀 + DB 마이그레이션 backward 실행. 롤백 후에도 cycles 복구 절차가 정상 동작해야 함

### 11-2.14 부분 체결 reconcile + 주문 유형 정책표

**부분 체결이란**: 100주를 사려고 주문해도 *그 가격에 팔겠다는 물량이 그 순간 60주뿐이면 60주만 사지고 40주는 안 되는 것* — 주문 수량 중 일부만 거래가 성립하는 일이다. 호가창에 반대편 물량이 부족하면 흔히 이렇게 쪼개져 체결된다.

**문제 시나리오**: 주문에 *즉시 체결 안 된 잔량을 어떻게 할지*를 지정하지 않으면(IOC = 되는 만큼만 체결하고 나머지 즉시 취소, FOK = 전량 즉시 안 되면 전부 취소), KIS는 잔량을 *대기 주문*으로 남겨 시차를 두고 나눠 체결할 수 있다. 이 *예상 못한 부분 체결*을 방치하면 *얼마를 샀는지(사이징)·언제 다 팔렸는지(청산)*의 정밀도가 망가진다.

**주문 유형 정책표** (외부 API 레퍼런스 문서의 ORD_DVSN 전체 표를 본 결정에 사용):


| 용도                | ORD_DVSN                | 이유                                                                |
| ----------------- | ----------------------- | ----------------------------------------------------------------- |
| **진입 (신규 매수)**    | `11` IOC지정가             | 즉시 체결 가능량만 진입, 잔여 자동 취소. *전량 체결 보장*보다 *체결 가능량 즉시 진입*이 사이즈 정밀도에 유리 |
| **청산 (논지 무효 강제)** | `13` IOC시장가             | 무조건 즉시 청산 의도, 잔여 취소                                               |
| **손절 (스톱지정가)**    | `22` 스톱지정가              | 손절 설계 결정. CNDT_PRIC 트리거 도달 시 KIS가 자동 발송                           |
| **부분 익절 (+1.5R)** | `00` 지정가 또는 `11` IOC지정가 | 호가창 깊이에 따라 결정                                                     |
| **트레일링 정정**       | 정정취소 API (`TTTC0013U`)  | 기존 스톱을 정정                                                         |


**부분 체결 처리**:

- IOC가 잔여를 자동 취소하므로 보통 발생하지 않으나, 진입 시 *부분 체결만 됐을 경우* SQLite `trades`에 *원주문 1행 + 체결 N행*으로 분할 적재. `cycles.status`는 `recorded`로 정상 종료
- 청산 시 부분 체결 잔여가 발생하면(예: 손절 스톱지정가 일부만 체결) 다음 사이클(또는 즉시 이벤트 트리거)에서 *잔여를 시장가 IOC로 강제 청산*

### 11-2.15 분기별 운영 점검 체크리스트

코드로 자동화하기 어려운 *수동 점검* 항목. 분기마다 한 번 운영자(=본인)가 직접 확인:

- ☐ **백업 복원 리허설**: 백업에서 복원해 사이클 1회 정상 동작
- ☐ **KIS 모의계좌 만료일 갱신**: D-30 알람 도착 여부 확인 + 갱신 작업
- ☐ **시크릿 회전**: KIS App Key/Secret 6개월 회전
- ☐ **모드 갭 측정값 재산정**: 동일 작은 주문을 모의·실전에 송출해 체결가 차이 재측정
- ☐ **외부 API 레퍼런스 문서 갱신**: KIS·yfinance·FRED·DART의 사양 변경·신규 엔드포인트 반영
- ☐ **세후 자본곡선 export**: `trades`·`outcomes` CSV export, 양도세·금소세 신고 자료 별도 보관
- ☐ **로그·SQLite 디스크 사용량 점검**: 80% 이하인가
- ☐ **heartbeat·알림 채널 동작 확인**: 의도적으로 ping을 누락시켜 알림이 오는지 검증
- ☐ **NTP 동기화 정확도**: 오차 ±1초 이내인가
- ☐ **백업 출처 동작 확인**: 1차 출처를 일부러 차단해 2차·3차 폴백이 실제로 도는지 검증

### 11-2.16 본 장의 한계와 단계적 도입

- 본 장의 18개 항목을 *Phase 0에서 한 번에 모두 구현*할 필요는 없다. 그러나 *원칙*은 Phase 0에 모두 박혀 있어야 한다(설계 단일 소스 역할)
- 구현 우선순위(로드맵 Phase와 정합): **(1차) idempotency · SQLite WAL · 시크릿 · 시간대 정규화** = Phase 0 필수. (2차) KIS 운영·heartbeat·백업 = Phase 1. (3차) "if 실전" 분기 금지 CI·마이그레이션·주문 유형 정책표 = Phase 2~3. (4차) 분기 운영 점검 = Phase 7 모의 검증 시작 시점
- 본 장 항목 누락이 *기능 합격*은 줄 수 있어도 *전략 합격*과 *장기 무인 운영*을 깨뜨린다는 점을 잊지 않는다. 1인 운영의 가장 큰 적은 *알고리즘이 아니라 silent failure*다.

### 11-2.17 자본시장법 게이트 — 외부 노출 금지

**문제 시나리오**: 본 시스템은 *1인 자기자본* 운영이라 자본시장법상 합법이다. 그러나 *시스템 결과·수익률·결정 로그를 외부에 노출*(블로그·구독제·SNS 공유·"포트폴리오 따라하기" 같은 형태)하면 *투자자문업·일임업 무인가 영업*에 해당해 자본시장법 §6 위반 위험이 생긴다. 시스템 자체보다 *운영자의 부주의*가 법적 리스크의 주된 경로다. (본 절은 신뢰성보다 *법무·컴플라이언스* 성격이나, "외부 노출 금지"가 곧 운영 규율이라 운영 단일 소스에 둔다.)

**원칙**:

- **외부 공개 노출 금지**: 결정·수익률·교훈·체결을 인터넷 공개 공간에 *공개 형태*로 노출하지 않는다. 블로그·SNS·유튜브·GitHub 공개 저장소·구독제 서비스 일체 금지
- **공유 의도 없는 자료도 보안**: 스크린샷·노트북 화면이 *우연히* 노출될 수 있으므로 작업 공간 보안(잠금화면·외부 발표 시 가림)은 운영자 본인 책임
- **회색지대 경계**: "기술 블로그에 *시스템 설계*만 공유"는 일반적으로 OK(투자자문 아님). 그러나 *실거래 수익률·구체 종목·매매 시점*을 공유하면 위반 가능성 — *수치를 가린 설계 추상화*까지만 외부 공유 허용
- **법적 자문 면책**: 본 절은 *운영 원칙 명시*일 뿐 법률 자문이 아니다. 실제 외부 노출 결정은 *자본시장법 전문 변호사 사전 상담 후*에만 진행. 본 절의 가이드 위반으로 인한 법적 책임은 운영자 본인에게 있다
- **저장소 정책**: 코드 자체는 비공개 저장소 권장. 설계 문서(본 계획서)도 *실거래 결과·실제 수익률 수치를 포함하지 않는 한* 공개 가능. *데이터·DB 파일·시크릿·로그 디렉터리*는 `.gitignore`에 강제 등록(시크릿 관리 원칙과 정합)

### 11-2.18 외부 데이터 출처 백업 경로

**문제 시나리오**: 데이터 소스 라이브러리(`yfinance`·`fredapi`·`OpenDartReader`)는 모두 *단일 출처* 의존이다. yfinance는 야후 비공식 크롤이라 차단·구조 변경 시 며칠~몇 주 중단되고, DART API는 분당 1,000회·일일 10,000회 한도, FRED는 美연준 서비스라 단기 장애 가능. 단일 출처 의존 시 *그 출처 장애가 매매 사이클 정지*로 직결된다.

**원칙 — 출처별 백업 경로 사전 등록**:


| 1차 출처                           | 2차 백업                           | 3차 백업                       |
| ------------------------------- | ------------------------------- | --------------------------- |
| **yfinance** (미국 지수·환율)         | Stooq (`pandas_datareader` 어댑터) | Investing.com 스크래핑 (최후 수단)  |
| **FRED** (美10Y·美 기준금리)          | 백업 없음 — 결측 시 매크로 비중↓(4.0)      | (한은 기준금리는 이벤트성→마지막 값 유지)     |
| **DART** (공시)                   | KIND(한국거래소 공시)                  | KRX 정보데이터시스템                |
| **KIS** (시세·수급)                 | KRX 정보데이터시스템 (유료)               | `pykrx` (KRX 공개 데이터 비공식 래퍼) |
| **exchange_calendars** (휴장 캘린더) | KRX 휴장공지 RSS                    | KIS 영업일 API                 |


**구현 정책**:

- *출처별 데이터 어댑터*가 *1차 → 2차 → 3차* 순으로 시도. 1차 실패(5xx·timeout·구조 변경 감지) 시 2차로 자동 폴백
- 출처별 어댑터는 *동일 데이터 스키마*를 반환해 호출 측은 *어느 출처를 썼는지 모른다*. 호출 시 *사용된 출처 라벨*을 함께 반환해 `data_vintage` 컬럼에 적재 — 사후 감사 시 *어느 사이클이 백업 출처로 돌았는지* 추적 가능
- 폴백 발생 시 heartbeat 채널로 *경보*(시스템 정지는 아니지만 *주의* 신호). 3차 출처까지 도달하면 *경보 우선순위 상향* + 다음 사이클부터 1차 출처 자동 재시도(복구 후 자동 1차 복귀)
- 분기별 운영 점검 체크리스트에 *백업 출처 동작 확인* 추가 — 폴백 경로가 *실제로* 동작하는지 분기별 테스트(1차를 일부러 차단 후 2차·3차 폴백 동작 검증). 백업 경로가 코드에만 있고 실제로 안 도는 사고를 차단

## 11-3. LLM 운영 정책 (재시도·캐싱·부분 실패·환각 통제)

> 운영 신뢰성 장(11-2)이 *시스템 운영*의 단일 소스라면, 본 장은 *LLM 호출 운영*의 단일 소스다. LangChain류 프레임워크를 회피한 결정의 *대가*로 직접 짜야 하는 보일러플레이트가 있다. 그 정책을 본 장에 모아, 기술스택의 LLM 라이브러리 선택과 에이전트 결정 룰의 *어떻게*를 명시한다.

### 11-3.1 재시도·백오프 정책 (429·529·5xx)

LLM API는 *일시적 오류*와 *영구적 오류*가 섞여 온다. 같은 정책으로 다루면 영구 오류에 무한 재시도 비용이 들거나 일시 오류에 즉시 포기해 사이클이 깨진다. 종류별로 다른 정책을 박는다.

- **429 Too Many Requests** (rate limit): 지수 백오프(1초→2초→4초→8초) 최대 4회. Anthropic은 `retry-after` 헤더를 줄 때가 있어 *그 값을 우선*하고, 없으면 지수 백오프
- **529 Overloaded** (Anthropic 일시 과부하): 동일 정책 — 지수 백오프 4회
- **500·502·503·504**: 지수 백오프 3회. 모두 실패 시 *해당 LLM 호출*만 실패 처리(분석가라면 "응답 없음", 결정자라면 사이클 중단)
- **400·401·403·404** (영구 오류): 재시도 *금지*. 401·403은 즉시 SafeStop(키 문제). 400·404는 코드 버그라 즉시 사이클 중단 + 경보
- 모든 재시도는 `structlog`로 *호출별 재시도 횟수·총 지연·최종 결과* 로그 적재
- 결정자 호출의 재시도는 *매 시도마다 다른 출력*을 낼 수 있으나 idempotency 위반 위험은 없다 — 결정자 출력은 SQLite에 *최종 1건*만 기록되므로

### 11-3.2 동시성 한도 (asyncio.Semaphore)

린 구성에선 사이클당 LLM 호출이 *뉴스 1 + 결정 1*뿐이라 동시성 압력이 작다(과거 "분석가 5명 동시 호출" 가정은 폐기). 그래도 이벤트 트리거가 겹치거나 향후 에이전트가 늘 때를 대비해 한도는 유지한다.

- `AsyncAnthropic` 클라이언트 1개에 *프로젝트 전역 Semaphore*(기본 동시 4, 설정값). 뉴스·결정·회고(도입 시)가 모두 이 한도를 통과해야 호출
- 사이클 내부는 *뉴스 1 → 결정 1* **순차**(뉴스 판단이 결정 입력이라 본래 직렬), 회고는 다른 시점. 즉 사이클 내 피크 동시 호출은 사실상 1~2 — `asyncio.gather` 병렬화의 필요성도 작다
- Anthropic plan별 한도(tier별 RPM·TPM)는 *외부 설정*으로 외부화
- 한도 초과 시 *앞의 재시도 정책* + 내부 우선순위(결정 > 뉴스 > 회고)

### 11-3.3 JSON 스키마 검증 + Partial JSON 처리

LLM이 JSON 출력을 *부분적으로* 망가뜨릴 수 있다 — 트리밍된 응답, 잘못된 escape, 누락 필드 등. *절반 결정*을 *결정*으로 채택하면 결정 룰 전체가 무너진다.

**정책 — pydantic v2 검증 단일 진입점, 4단계 폴백**:

1. **1차: 마크다운 코드블록 추출** — LLM이 응답을 코드블록으로 감싼 경우 정규식으로 JSON만 추출
2. **2차: JSON repair** — `json-repair` 같은 라이브러리로 *최소 수정*만 (콤마 누락·따옴표 정리). *내용 추정 금지* (필드 값을 추측해서 채우지 않음)
3. **3차: 동일 입력으로 재호출 1회** — `temperature=0.1`로 낮춰 *더 결정론에 가깝게* 재시도
4. **4차: 그래도 실패 시 그 호출은 실패 처리** — 결정자면 사이클 중단, 분석가면 *부분 실패 룰* 적용

**Partial JSON 허용 금지**: 필수 필드 누락된 부분 파싱을 *유효 응답으로 채택하지 않는다*. "절반 결정"이 "결정 안 함"보다 위험.

검증 실패 통계는 `llm_calls`의 `parse_status` 컬럼(`ok`/`repaired`/`retried`/`failed`)에 적재 → `agent_role`로 GROUP BY 하면 자주 실패하는 분석가·프롬프트가 그대로 나온다(파싱 실패는 견해 행 자체를 못 만들므로 호출 단위에 둔다 — *데이터 모델 장*과 정합).

### 11-3.4 프롬프트 캐싱 (5분 TTL · 1024 토큰 최소 청크)

Anthropic 프롬프트 캐싱은 시스템 프롬프트 캐시 적중 시 입력 토큰 비용을 ~90% 절감한다. **단 린 구성에선 캐싱 효과가 작다 (정직 고지)** — 과거 큰 이득은 *분석가 5명이 사이클 안에서 같은 프롬프트를 공유*하던 데서 나왔는데, 이제 뉴스·결정은 *서로 다른 시스템 프롬프트를 각 1회씩만* 쓰므로 사이클 내부 재사용이 거의 없고, 사이클 간격(4.5h)은 5분 TTL을 훨씬 초과해 사이클 간 캐시도 없다. 그래도 입력 컨텍스트(정적 룰·스키마·시스템 프롬프트)가 호출당 토큰의 큰 부분이라 *적용 자체는 손해가 없어* 유지한다(비용 절대액이 이미 작아 캐싱은 보조 최적화).

**정책 — 명시적 `cache_control` 경계**:

- **뉴스·결정 시스템 프롬프트**: 각자 사이클 간 변하지 않는 부분 → `cache_control: {type: "ephemeral"}` 적용
- **도구 정의 + 정적 컨텍스트**(예: 한국시장 규제·세금 룰)도 캐시 경계 안에
- **동적 데이터(가격·수급·뉴스 헤드라인·교훈)는 캐시 *밖***. 매 사이클 새 데이터라 적중률 0
- **5분 TTL의 영향**: 정기 사이클 4.5h 간격 ≫ TTL이라 *사이클 간 적중은 사실상 0*. 이벤트 트리거가 5분 내 연달아 도는 드문 경우에만 사이클 간 적중이 생긴다 — 받아들임
- **최소 청크 1024 토큰**: 시스템 프롬프트가 이 이상이어야 캐시 의미. 각 에이전트별 2~5K로 충분
- **캐시 적중률 KPI**: `cache_read_input_tokens` 비율을 로그·일간 집계해 *추적*하되, 린 구성에선 목표치를 높게 잡지 않는다(구조상 낮은 게 정상)

### 11-3.5 부분 실패 운영 메커니즘

린 구성은 분석가가 뉴스 1명뿐이라 과거 "다수결(4명 이상 응답)" 룰은 적용되지 않는다. 호출별로 다룬다:

- 뉴스·결정 호출에 *개별 타임아웃*(뉴스 45초·결정 60초, 설정값) + 앞의 재시도 정책. 실패 판정 시 종류별로:
- **뉴스 분석가 실패**(타임아웃·반복 파싱 실패): 사이클을 죽이지 않고 *촉매 입력 없이* 진행 — 코드 산출 숫자 점수만으로 결정자가 판단하되 *촉매 부재를 입력에 명시해 확신도를 낮춘다*(2.5절 촉매 인지 한계와 일관). config로 "뉴스 필수" 설정 시엔 그 사이클 무거래
- **결정 에이전트 실패**: 결정 주체가 없으므로 그 사이클은 *무거래* 자동 처리(`no_trade_reason="decider_unavailable"`). (B안 운영 = 결정자 제거 구성이면 코드 규칙이 결정하므로 이 실패와 무관)
- SafeStop은 *시스템 오류*에 reserved하고, *LLM 일시 가용성 문제*는 무거래로 흡수
- 실패 통계는 `cycles.failed_agents`·`llm_calls.parse_status`에 적재 → 사후 점검

### 11-3.6 모델 deprecation 정책

Claude 모델은 주기적으로 deprecation된다(예: Sonnet 4 → 4.5 → 4.6). 모델 ID를 코드에 박으면 어느 날 호출이 실패하기 시작한다. 또한 *모델 버전이 바뀌면 calibration 보정 통계의 유효성이 흔들린다* — 다른 모델은 다른 응답 패턴이라 과거 적중률을 그대로 믿으면 학습 루프가 망가진다.

**정책**:

- 모델 ID는 *외부 설정*으로 외부화. 코드에 박지 않음
- 모델 별칭 사용 vs 고정 ID — 둘 다 트레이드오프가 있어 *분기별 운영 점검* 항목. 별칭은 *예상치 못한 업그레이드 위험*, 고정 ID는 *deprecation 사고 위험*
- **모델 변경 시 calibration 정책**: 모델 변경 시점부터 *기존 보정 통계에 시간 가중치 추가 패널티*(예: 0.5배) 적용 — *학습 루프의 시간 가중치 정책*과 별개로. 변경 후 N=20 표본이 새로 쌓이면 패널티 자동 해제
- `agent_predictions`에 `model_version` 컬럼 추가해 어느 모델이 어떤 응답을 냈는지 사후 감사 가능. *모델별로 따로 calibration*도 표본이 쌓이면 자동 도입(*학습 루프의 세션 차원 통계 도입 임계*와 동일 메커니즘)

### 11-3.7 로깅·관찰 가능성 (`llm_calls` 테이블)

LLM 호출은 *비용·지연·환각*의 1차 자료다. SQLite `llm_calls` 테이블에 호출당 1행씩 적재:

- 요청 페이로드(시스템 프롬프트는 해시·길이만, 동적 본문은 전체)
- 응답 페이로드(전체)
- 토큰 사용량(`input_tokens`·`output_tokens`·`cache_creation_input_tokens`·`cache_read_input_tokens`) — Anthropic SDK 응답의 `usage` 객체에서 호출마다 실측(추정 아님)
- **호출당 비용**(`cost_usd`·`cost_krw`): 위 네 토큰 수에 *그 시점 모델별 단가*를 곱해 한 호출의 비용을 적재. 단가는 코드에 박지 않고 *날짜별·모델별 외부 설정*(모델별 input·output·cache_read·cache_write 단가)으로 분리 — 단가가 바뀌면 설정만 고치고, 과거 비용은 *그 시점 단가*로 정확히 재현(거래세율 설정과 같은 패턴). USD→원화 환산은 이미 수집하는 USD/KRW로. 캐시 읽기 토큰은 정가의 약 1/10이라 비용 대부분은 *새 입력 + 출력*에 붙는다(*프롬프트 캐싱*이 이 항을 직접 줄임)
- 지연(요청~응답)
- 모델 ID + 모델 버전
- 재시도 횟수 + 최종 결과(`ok`/`repaired`/`retried`/`failed`)
- `cycle_id` 참조 (`cycles` 테이블과 연결)

이 테이블은 *학습 데이터 모델*에도 명시한다. *직접 호출이면 페이로드·응답을 같은 화면에서 본다*는 설계 원칙의 구현체다.

토큰·비용 일간/월간 집계 → 대시보드에 노출, 일간 비용 한도 80% 도달 시 heartbeat 채널로 경보. **이 비용은 *순이익(net) KPI*에서 차감된다** — LLM 운영비는 거래당이 아니라 *사이클 고정비*다(스크리닝 통과 종목을 분석만 하고 안 사도 토큰은 쓰고, 무거래 사이클도 비용이 발생한다). 그래서 거래세·슬리피지처럼 개별 체결에 귀속하지 않고 *기간(일/월) 단위로 자본곡선에서 뺀다*. 집계는 분석가별·사이클별로도 분해 가능(`agent_role`·`cycle_id`로 GROUP BY) — 어느 전문가가 토큰을 제일 많이 먹는지, 거래 1건당 LLM비용이 얼마인지 추적. LLM 호출 *지연 분포*도 추적 — p50·p95·p99. p95 > 20초 지속이면 모델 측 이슈 또는 우리 프롬프트 크기 점검.

### 11-3.8 본 장과 다른 장의 관계 (단일 소스 매트릭스)

- **LLM 라이브러리 선택**(`anthropic` SDK·`asyncio`) = *기술스택* 장
- **분석가·결정자·회고의 역할과 결정 룰** = *에이전트* 장
- **LLM 운영 메커니즘(재시도·캐싱·부분 실패·환각 통제)** = *본 장*
- **시스템 운영 신뢰성(idempotency·SQLite·시크릿·heartbeat 등)** = *운영 신뢰성* 장
- **비용 통제 원칙** = *비용 통제* 장 (본 장의 캐싱·모델 차등이 그 *구현 정책*)

본 장이 *비용 통제* 원칙과 충돌하면 *비용 통제가 원칙, 본 장이 구현*이다 — 비용 한도가 깨질 위험이 보이면 *정기 2회 + 이벤트 트리거·모델 차등* 같은 상위 정책에 우선 손댄다(이벤트 트리거 일일 상한 축소 등).
