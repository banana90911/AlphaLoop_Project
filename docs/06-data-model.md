## 6. 기억·학습 데이터 모델 (학습이 무엇을·어디에·어떻게 저장하나)

> "학습" = 모델 가중치 재학습이 아니라 SQLite에 결정·결과·교훈·통계를 쌓고 다음 결정에 인출·주입하는 폐(閉)루프. **모든 통계 산출은 결정론 코드(SQL 집계)가 하며 LLM은 관여하지 않는다**.

**저장소**: 데이터베이스 = **SQLite 파일 1개**(별도 DB 서버 없음). 동일 파일을 대시보드가 읽음(매매 코어와 SQLite로만 결합 → 가역적).

- 파일 하나가 엑셀 워크북이라면 테이블은 그 안의 시트들(`decisions`·`trades`·`lessons`…)이고, 레짐은 별도 시트(테이블)가 아니라 `decisions`·`lessons` 시트의 한 컬럼(태그)이다. 결정 1건을 내리면 `decisions` 시트에 1행이 추가되고 그 행의 한 칸이 레짐이다.

**객체 카탈로그**

객체를 성격별로 묶으면 다음과 같다. 컬럼 상세는 바로 아래 *객체별 컬럼 정의*에 있다.

| 묶음             | 객체                                                   | 한 줄 성격                      |
| -------------- | ---------------------------------------------------- | --------------------------- |
| 운영(복구·감사·비용)   | `cycles`                                             | 사이클 1회의 상태 머신(idempotency)  |
| 〃              | `trigger_events`                                     | 이벤트 트리거 감지(발동·미발동 모두)       |
| 〃              | `safe_stop_events`                                   | SafeStop 발생·해제 감사           |
| 〃              | `llm_calls`                                          | LLM 호출 1건의 감사·토큰·비용         |
| 보유·체결·결과(사실)   | `positions`                                          | 내부 보유 스냅샷(정합성 게이트·진입↔청산 연결) |
| 〃              | `frozen_positions`                                   | 거래정지 종목 MTM 동결              |
| 〃              | `trades`                                             | KIS 주문·체결 1건                |
| 〃              | `outcomes`                                           | 포지션 청산 실현손익                 |
| 의사결정·학습(정성·정량) | `decisions`                                          | decider 제안주문 1건             |
| 〃              | `agent_predictions`                                  | 거래 종목별 LLM 에이전트 견해 + 사후 적중·귀속 (린 구성: 뉴스·결정자) |
| 〃              | `lessons`                                            | 회고 교훈 + 메타                  |
| 〃              | `shadow_outcomes`                                    | 거부·미진입 반사실                  |
| 집계(통계)         | `calibration`(뷰)·`calibration_cold`·`shadow_summary` | 보정·장기 보존·shadow 압축 집계       |

> 아래 컬럼은 **논리 수준**이다. 물리 타입(INTEGER/TEXT/REAL)·NULL 허용·인덱스·KIS/DART 응답 필드와의 정확한 매핑·enum 확정값은 착수 시 *API 명세(외부 API 레퍼런스 문서)*에서 확정한다. 본 절은 "어떤 객체에 어떤 컬럼이 왜 있는가"의 단일 소스다.
>
> **공통 규칙**: ① PK는 각 객체의 `*_id`. ② `source TEXT CHECK(source IN ('backtest','paper','live'))`는 학습·체결 계열(`decisions`·`trades`·`outcomes`·`agent_predictions`·`shadow_outcomes`)에 둔다(모드 라벨, 아래 *모드 라벨* 참조). `cycles`·`positions`·`frozen_positions`·`lessons`는 자체 `source`를 두지 않는다 — 사이클·보유·동결의 모드는 산출물(`decisions`·`trades`)을 FK로 따라가고, `lessons`는 모드를 가로질러 집계한다. ③ 시각 컬럼은 UTC 저장·KST 표시.

**객체별 컬럼 정의**

*`cycles` — 사이클 상태 머신*

| 컬럼                                       | 의미                                                                                          |
| ---------------------------------------- | ------------------------------------------------------------------------------------------- |
| `cycle_id` (PK)                          | timestamp 기반 발급. 모든 사이클 산출물의 부모 키                                                           |
| `status`                                 | `intent`→`ordering`→`recorded`, 실패 시 `failed` (복구 상태 머신)                                    |
| `trigger_type`                           | `scheduled`(정기 2회) / `event`(이벤트 트리거)                                                       |
| `trigger_event_id` (FK→`trigger_events`) | 이벤트 사이클을 발동시킨 감지 행. 정기 사이클은 NULL. *트리거 상세(종류·종목·기준가)는 `trigger_events`에 두고 여기선 참조만 — 중복 제거* |
| `started_at`·`finished_at`               | 시작·정상 종료 시각. `session_label`(morning/afternoon)은 `started_at`에서 유도되며 `decisions`에 적재        |

*`trigger_events` — 트리거 감지(발동·미발동 모두)*

미발동 감지도 남겨 *가드가 과한지(중요 순간 놓침)*를 사후 검증한다 — `shadow_outcomes`와 같은 반사실 진단이자 트리거 임계(±X%) 튜닝의 1차 자료.

| 컬럼                | 의미                                                               |
| ----------------- | ---------------------------------------------------------------- |
| `trigger_id` (PK) |                                                                  |
| `detected_at`     | 감지 시점                                                            |
| `subtype`         | `dart`(재료성 공시) / `price_move`(±X% 급변). 향후 `vi`·`circuit_breaker` |
| `symbol`          | 보유 종목(이벤트 트리거는 보유 방어 전용)                                         |
| `detail`          | 예: `+7.2%`, 공시 접수번호                                              |
| `reference_price` | 직전 사이클이 그 종목을 본 기준가(±X% 판정 기준, 수정주가)                             |
| `fired`           | 발동 여부                                                            |
| `skip_reason`     | 미발동 사유(10분 가드·일일 상한 도달 등). `fired=false`일 때                      |

| `released_at`    | 해제 시점(해제 전 NULL)                                       |
| `released_by`    | 해제자(모델 이상행동·잔고불일치·데이터오류는 사람 개입 필수)                     |
| `release_reason` | 해제 사유                                                  |
| `cycle_id` (FK)  | 연관 사이클(있을 시)                                           |

| 컬럼                                                                         | 의미                                                                                                                                               |
| -------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |

*`positions` — 내부 보유 스냅샷 (신규)*

> 선행 게이트가 *KIS 실잔고와 대조*하는 우리 측 기록이자, 진입 결정(`decisions`)↔청산 결과(`outcomes`)를 잇는 다리다. KIS 라이브 조회는 *현재 진실*이고 본 객체는 *우리가 기억하는 상태* — 둘을 대조해 어긋나면 그 사이클을 멈춘다.

| 컬럼                                   | 의미                                 |
| ------------------------------------ | ---------------------------------- |
| `position_id` (PK)                   |                                    |
| `symbol`                             |                                    |
| `qty`·`avg_price`                    | 보유 수량·평단가                          |
| `sector`                             | KRX 업종(섹터 집중도 한도·교체매매용)            |
| `entry_decision_id` (FK→`decisions`) | 이 포지션을 연 진입 결정                     |
| `current_stop_price`                 | 현재 KIS에 상주하는 스톱지정가 트리거(매 사이클 재동기화) |
| `status`                             | `open`/`closed`                    |
| `opened_at`·`updated_at`             | 최초 진입·마지막 동기화 시각                   |

*`frozen_positions` — 거래정지 종목 MTM 동결 (신규)*

> 거래정지·관리종목·투자경고/위험 보유 종목을 *정지 직전 종가로 얼려* 자본곡선·KPI가 흔들리지 않게 한다. 재개 시 실제 시가로 재평가.

| 컬럼                                      | 의미                                       |
| --------------------------------------- | ---------------------------------------- |
| `frozen_id` (PK)                        |                                          |
| `position_id` (FK→`positions`)·`symbol` |                                          |
| `frozen_at`                             | 동결 시점(정지 직전)                             |
| `last_valid_price`                      | 마지막 유효 시가(MTM 동결값 — KIS가 시가를 안 주는 경우 대비) |
| `halt_reason`                           | 거래정지 / 관리종목 / 투자경고·위험                    |
| `released_at`                           | 재개 시점(재개 전 NULL)                         |

`decisions` *— decider 제안주문 1건 (출력 스키마와 1:1)*

| ------------------------------------- | ---------------------------------------------------------------------------- |
| `decision_id` (PK)                    |                                                                              |
| `cycle_id` (FK)                       | 레짐은 이 결정의 컬럼(`regime_tag`)에, 세션은 `session_label`에 둔다(아래)                     |
| `symbol`                              | no_trade 시 후보 종목(또는 NULL)                                                    |
| `action`                              | `buy`/`sell`/`trim`(분할청산)/`trail`(트레일링)/`exit`(전량청산)/`no_trade`              |
| `side`                                | 매수/매도                                                                        |

| `regime_tag`                          | 레짐 스냅샷 태그(코드가 매 사이클 산출, 결정 시점 고정 — 사후 회고가 국면을 알게)                            |
| `session_label`                       | `morning`/`afternoon`(보정 통계의 세션 차원 단일 소스)                                    |
| `source`·`decided_at`                 |                                                                              |

*`agent_predictions` — LLM 에이전트 견해 + 사후 평가, (cycle, symbol, agent) 단위*

> **단위 결정**: decider가 실제 결정을 내린 *종목마다*, 그 종목을 본 LLM 에이전트의 견해를 1행씩 적재한다(린 구성 = `cycle × 거래종목 × {뉴스, 결정자}`; 풀 구성으로 분석가를 재도입하면 그만큼 행이 는다). 워치리스트 전체가 아니라 *결정된 종목*만 기록해 공헌도·보정이 그 종목의 `outcomes`에 정확히 연결되고 빈 행이 쌓이지 않는다. no_trade도 결정이므로 그 종목의 견해는 적재되며 사후 평가는 `shadow_outcomes`(반사실)와 연결된다. 이벤트 캘린더 신호는 코드 플래그라 예측 행이 아니라 `decisions.context_snapshot`에 기록 — 공헌도 분해 대상은 LLM 에이전트뿐이다(린 구성에선 뉴스·결정자, 칸이 5배 적어 보정통계가 더 빨리 의미를 가진다 — 7.19).

| 컬럼                                                      | 의미                                                                                        |
| ------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| `prediction_id` (PK)                                    |                                                                                           |
| `cycle_id` (FK)·`decision_id` (FK→`decisions`)·`symbol` | 귀속 대상. 레짐·세션은 `decision_id`→`decisions`로 인용                                               |
| `agent_role`                                            | 린 구성: `catalyst`(뉴스)/`decider`. 회고 도입 시 `reflection`. enum 확장 가능                          |
| `view`                                                  | 강세/약세/중립(방향)                                                                              |
| `confidence`                                            | 0-1 자기신고(스키마 게이트 0-1·필수 검증 통과 후 적재)                                                       |
| `key_signals`·`key_risks`                               | 신호·리스크 배열(촉매·펀더멘털은 신호별 `news`/`fundamental` 출처 라벨)                                        |
| `rationale`                                             | 근거(Warm 진입 시 50자 축약)                                                                      |
| `model_version`                                         | 모델별 보정용. `llm_calls`에도 있으나 *의도적 복제* 보존 — 페이로드가 먼저 압축·아카이브돼도 Warm 단계 보정이 모델별로 재현돼야 함(17.6) |
| `correct`                                               | 적중 1/0 — 보유기간 비용차감 수익률 부호 vs `view` 일치(청산/N영업일 평가 시 갱신)                                   |
| `tentative`                                             | 잠정 라벨(N영업일 중간평가 True → 청산 시 확정 False)                                                     |
| `attribution_score`·`realized_pnl_attribution`          | 공헌도 점수(누적)·실현손익 귀속분                                                                       |
| `source`                                                |                                                                                           |

*`lessons` — 회고 교훈 + 메타*

| 컬럼                             | 의미                                                         |
| ------------------------------ | ---------------------------------------------------------- |

**[쉬운 설명] 레짐 ↔ 테이블 ↔ 주입 관계**

- **레짐은 테이블이 아니다**: 레짐은 코드가 시장데이터로 계산하는 "오늘 시장 날씨" 라벨이다. `decisions`·`lessons` 행에 **태그 컬럼으로 박혀** 저장된다 — 레짐이 테이블 *안으로* 들어가는 것이지, 테이블이 레짐 안에 있는 게 아니다.
- **레짐이 사는 3곳**: ① 계산(매 사이클 Tier0/1 데이터로) → ② 저장(`decisions`, 청산 후 `lessons` 태그) → ③ 사용(인출 때 "오늘 레짐 vs 과거 교훈의 레짐" 비교로 닮은 교훈 선별).
- **무엇이 컨텍스트로 주입되나 — 원시 테이블 통째가 아니다**: 결정자 프롬프트에 들어가는 것은 [6 전문가 분석] + [스코어링으로 *고른* `lessons`] + [`calibration` 통계] + [리스크·포트폴리오·보유] + [오늘 레짐·시장상황]. `trades`·`agent_predictions`·`shadow_outcomes` 원시 테이블 전체는 안 들어간다 — 코드가 압축·집계해 만든 `lessons`·`calibration`만 주입(데이터 압축 원칙).
- **시점별 저장**: 무엇이 언제 어느 테이블에 쓰이는지는 위 **"저장 타임라인"** 표 참조(단일 소스).

**보존·압축 정책 — 시간 가중·계층 보존 (시계열 비정상성 대응)**
시계열 데이터는 이미지 데이터와 다르게 *"데이터를 만들어내는 규칙 자체"*가 시간이 지나면 변한다. 같은 "강세장" 라벨이라도 2017-2018 반도체 슈퍼사이클, 2020-2021 코로나 유동성 장세, 2022 금리 급등 성장주 붕괴, 2023~2024 AI 1차 테마는 모두 *다른 시장*이다 — 시장 참여자·자금 흐름·매크로 환경이 다르기 때문(통계학 용어로 *비정상성*이 크다고 한다). 데이터를 무차별 누적하면 *오래된 패턴*이 *지금 결정*에 영향을 미쳐 짐이 되고, 반대로 너무 빨리 버리면 학습 신호가 약해진다. 그 균형을 잡는 것이 이 정책의 목적이다.

원칙은 세 가지를 합쳐서 운용한다.

- **① 시간 가중치 (지수 감쇠)** — 오래된 데이터를 *지우지 않고 영향력만 줄인다*. 음식점 리뷰로 비유하면 10년 전 리뷰를 *그대로 두되* 평균 별점 계산할 때 가중치 0.1만 주는 식이다.
  - 공식: `weight = exp(-λ × 경과일수)`. 반감기(half-life) 1년이면 1년 전 데이터 가중치 0.5, 3년 전 0.125, 5년 전 0.03. 보정 통계·공헌도·교훈 신뢰도 산출이 단순 평균이 아니라 *최근 가중 평균*으로 바뀐다 — numpy 한 줄(`np.average(values, weights=np.exp(-λ * ages))`) 추가로 끝. **기본 반감기는 1년에서 시작하되 한국 스윙에 너무 길 가능성이 있다**(통계 문서 *학습 루프 정직 고지* 참조). 백테스트로 *3개월·6개월·1년·2년* 네 가지 반감기를 워크포워드 비교해 OOS 성과가 가장 좋은 값으로 확정(14장 결정 항목).
- **② 만료 (hard cutoff)** — 시간 가중치만 두면 가중치가 영원히 0이 안 된다(5년 후에도 0.03 잔존). 그래서 *너무 오래되거나 도움 안 된 것*은 인출 후보에서 아예 제외한다.
  - `lessons.expiry` 도래 → 인출 후보에서 빠짐. 회고 에이전트가 작성 시점에 기본 만료일을 부여(기본 24개월).
  - `helped_score` 누적 임계 이하(예: −2) → 만료 가속(3개월 단축). *"교훈을 주입했는데 결과가 더 나빠진 경우"* 가 반복되면 그 교훈은 더 빨리 비활성된다.
  - 만료는 *삭제*가 아닌 *비활성* — 사후 분석·감사는 여전히 가능하며, 인출 스코어러가 후보로 안 부른다는 의미.
- **③ 계층 보존 (Hot · Warm · Cold · Archive)** — 시간대별로 *보존 형태*를 달리한다. 용량은 줄이되 학습 신호는 살린다. 도서관으로 비유하면 최근 책은 1층 열람실(즉시 접근), 오래된 책은 지하 서고(보관만), 아주 오래된 책은 마이크로필름(압축)으로 옮기는 식.

| 단계      | 시간 범위    | 보존 형태                                                                                       |
| ------- | -------- | ------------------------------------------------------------------------------------------- |
| Hot     | 최근 6개월   | 모든 raw 보존(`entry_thesis`·컨텍스트 스냅샷·전문가 의견 근거 전문). 디버깅·인출·통계 모두 가능                            |
| Warm    | 6개월 ~ 2년 | 컨텍스트 스냅샷 컬럼 NULL, 전문가 근거 텍스트 50자 축약. *결정·결과·view·confidence·레짐 태그*는 보존. 통계·인출 가능, 사후 디버깅 제한 |
| Cold    | 2년 ~ 5년  | 행 단위 raw 삭제. 월별 집계 통계(`전문가 × 레짐 × 적중률·표본수`)만 별도 테이블 `calibration_cold`에 누적 보존. 인출 후보 아님     |
| Archive | 5년 이상    | 백테스트 재생으로 복원 가능하므로 별도 파일(`archive/journal_YYYY.sqlite.gz`)에 압축 백업 후 메인 DB에서 제거              |

**테이블별 압축 방식** — 행당 크기와 재현 가능성에 따라 다르게 다룬다.

| 테이블                   | 행당 크기        | 압축 방식                                                                                                 |
| --------------------- | ------------ | ----------------------------------------------------------------------------------------------------- |
| `decisions`           | 큼 (논지·컨텍스트)  | Warm 진입 시 컨텍스트 스냅샷 컬럼 NULL. 결정 본문·`entry_thesis`·`session_label`·레짐 태그는 그대로 보존                        |
| `agent_predictions`   | 중간 (view·근거) | Warm 진입 시 `근거` 텍스트 50자 축약. `view·confidence·적중여부·model_version`은 그대로(모델별 보정이 Warm에서도 재현돼야 함)          |

**백테스트 시드와의 결합 — 학습 신호 부족 + 비정상성 동시 해결**
Phase 6 백테스트로 5~10년 과거 데이터를 재생해 학습 테이블을 *사전 시드*한다. 이때 시간 가중치가 자연스럽게 적용되어:

- 5년 전 시드 데이터 가중치 0.03 (거의 영향 없음)
- 1년 전 시드 가중치 0.5
- 6개월 전 시드 가중치 0.7

즉 *백테스트는 칸을 채워 게이트(n≥20)를 통과시키는 역할*만 하고, *실제 보정 가중치는 신선한 실거래 데이터가 주도*한다. 학습 신호 부족(소표본) 문제와 비정상성(낡은 패턴 영향) 문제를 한 번에 푼다. 모의 시작 시점에 보정 통계가 이미 의미 있는 상태로 출발 가능.

**모드 라벨 (`source` 컬럼) — 모의→실전 전환 시 학습 데이터 유지**
설계 불변식("모의/실전은 config 플래그 하나로 전환, SQLite는 유일한 공유점")에 따라 모의→실전 전환 시 SQLite 파일을 그대로 들고 간다. 다만 *데이터 신뢰도가 모드별로 다른* 부분이 있다 — 모의 환경은 슬리피지·갭이 단순화되는 경우가 많아 체결 가격·실현 손익의 정확도가 실전과 다를 수 있다. 그래서 학습·체결 계열 객체(`decisions`·`trades`·`outcomes`·`agent_predictions`·`shadow_outcomes`)에 다음 컬럼을 둔다(`cycles`·`positions`·`frozen_positions`는 자체 `source` 없이 산출물 FK로 모드가 따라오고, `lessons`는 모드를 가로질러 집계하므로 제외):

```sql

source TEXT NOT NULL CHECK(source IN ('backtest','paper','live'))

```

또한 **KIS API 자체의 모의 지원 범위가 한정적**이라는 제약도 같은 컬럼으로 흡수한다. KIS의 *실현손익 조회*·*기간별 손익 합산 조회*·*기간별 매매손익 현황 조회* 같은 KPI 산출용 API는 모두 *모의투자 미지원*이다(KIS 명세서 확인 결과 — 상세 매트릭스는 외부 API 레퍼런스 문서 참조). 즉 모의 환경에서는 KIS에서 KPI를 *받아올 수 없다*. 이 제약을 우회하면서 모의·실전을 *같은 코드 경로*로 운영하기 위한 결정은 다음과 같다.

**KPI 계산은 모드와 무관하게 항상 우리 코드가 `trades`·`outcomes` 테이블에서 직접 한다.** KIS의 실현손익·기간손익 조회 API는 사용하지 않는다(실전에서도 사용하지 않음). 이 결정의 이점은 두 가지다. ⓐ *모의/실전 동일 코드 경로 유지* — 3.3 설계 불변식("`if 실전 then ... else` 분기 금지")과 정합. KIS 모의 미지원 API에 의존했다면 실전 분기를 강제로 박아야 했을 것이다. ⓑ *산식이 우리 손에 있어 정정 자유* — 거래세율 변경·금융투자소득세 시뮬·세후 수익 계산·비용 모델 보정을 KIS 응답 포맷 변경과 무관하게 우리가 통제한다. KIS가 산식·라벨을 바꿔도 우리 KPI는 흔들리지 않는다.

집계 시 모드별 사용 정책:

- **KPI 산출 주체**: 모드 무관 — `eval/metrics.py`가 `trades`·`outcomes`에서 numpy/pandas로 직접 계산. KIS의 실현손익·기간손익 조회 API는 사용하지 않는다(위 결정).
- **교훈·보정 통계·공헌도**: 모든 source를 활용 (LLM 동작은 모드 무관, 학습 신호는 모드에 관계없이 가치 있음). 단 `paper`·`backtest`에는 시드 가중치(기본 0.5, config)를 곱해 실거래 신호가 주도하게 한다.
- **수익 KPI 표시(샤프·MDD·Profit Factor 등)**: 대시보드 기본은 `WHERE source = 'live'`로 필터링. 모의 KPI는 슬리피지·체결 단순화로 부풀려질 수 있어 본 지표로 쓰지 않는다. `paper`·`backtest` KPI는 *별도 비교 탭*에 표시해 모드 갭(모의가 실전보다 얼마나 낙관적인지)을 시각적으로 확인할 수 있게 한다.
- **모의 KPI 패널티 할인 (Go/No-Go 게이트용)**: Phase 7 모의 검증의 KPI를 Go/No-Go 게이트에 *직접* 사용하지 않는다. *측정된 모드 갭 비율*(5.2절 참조)만큼 할인한 값을 임계와 비교한다. 이는 모의 KPI가 실전 대비 부풀려진다는 사실(KIS 모의의 슬리피지 0 가정 등)을 정량적으로 반영하는 안전장치다.

**용량 추정 (5년 운영 후)**

- 압축 없음: 약 700MB
- 압축 적용: **약 100~200MB**

SQLite 단일 파일·맥미니 디스크에 부담 없는 수준이며 백업도 한 파일 복사로 끝난다.

**한계 — 정직 고지**

- *λ(감쇠율) 자체가 또 하나의 튜닝 파라미터*: 반감기가 1년이 맞는지 6개월·3년이 맞는지 정해야 한다(추후 결정 항목).
- *시장 구조 변화는 사후에야 안다*: 2022년 금리 급등 같은 큰 변화는 그 직후엔 안 보이고 1년쯤 지나야 보인다. 그래서 *지수 감쇠*가 *하드 컷오프*보다 안전 — 사람이 변화 시점 정할 필요 없이 자연스럽게 약해진다.
- *압축은 비가역적*: Cold 단계의 raw 삭제는 그 시점 컨텍스트를 영구 손실시킨다. Archive 단계의 압축 백업이 마지막 안전망.
- *계층 경계는 초기 추정치*: 6개월·2년·5년이 절대값이 아니다. 데이터 양·검색 성능을 보고 운영하며 조정.
