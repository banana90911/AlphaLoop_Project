## 9. 기술스택

"프레임워크 미사용·직접 호출" 원칙에 따라, 두꺼운 프레임워크는 의도적으로 배제하고 **단일 목적의 검증된 라이브러리만 직접 조립**한다. 표준 라이브러리(stdlib)로 충분한 경우는 외부 패키지를 더 얹지 않는다. 영역별 선택과 근거는 다음과 같다.

### 9.1 언어 — Python

트레이딩·LLM·과학계산 세 생태계를 동시에 가장 두텁게 받쳐주는 유일한 선택. KIS API·KRX 데이터·Anthropic SDK·`pandas`/`numpy`가 모두 1급으로 지원되어, 데이터 적재부터 LLM 호출, 백테스트까지 단일 언어 안에서 끝난다. 아래 모든 영역이 Python 단일 런타임 위에서 돌아간다는 점이 운영·디버깅·로깅의 일관성을 만든다.

### 9.2 LLM 호출 — `anthropic` 공식 SDK + `asyncio`

- `**anthropic`** — Claude API 호출. 뉴스·촉매 분석, 결정, 회고(도입 시) 모두 Claude 단일 공급자로 가능하므로 멀티공급자 프레임워크의 핵심 가치가 사라진다. `client.messages.create(...)` 한 호출에 보낸 파라미터와 받은 응답이 1:1 매핑되어 그대로 SQLite에 적재하면 감사·재현 요건이 끝난다. 프롬프트 캐싱(시스템 프롬프트·도구정의 캐시로 비용 절감), 구조화 JSON 출력, 도구 사용, 비동기(`AsyncAnthropic`)가 추가 래퍼 없이 SDK 표면에 그대로 노출된다.
- **모델 차등**: 결정 에이전트=Sonnet(Opus 불요 — 입력 대부분이 코드 숫자), 뉴스·촉매 분석=Haiku/Sonnet, 회고(도입 시)=Sonnet (3-2장 린 구성).
- `**asyncio`** (표준 라이브러리) — 여러 LLM 호출(이벤트 동시 처리·향후 에이전트 추가)을 묶을 때 `asyncio.gather(...)` 한 줄로 필요한 병렬도가 표현된다. 동시 호출 한도는 프로젝트 전역 `asyncio.Semaphore`(기본 4, 설정값)로 묶는다(11-3.2). 린 구성에선 사이클 내 호출이 *뉴스→결정* 1~2개뿐이고 본래 순차라 병렬 필요성 자체가 작다. 별도 그래프·실행기 라이브러리 불필요.
- **재시도·백오프 — 직접 구현(재시도 라이브러리 미채택)**: 429·529·5xx 일시 오류에 지수 백오프(1→2→4→8초), 영구 오류(400·401·403·404)는 즉시 중단, *주문 송출은 재시도 금지*라는 종류별 분기(17.1)를 `tenacity`·`backoff` 같은 라이브러리 없이 직접 짠다. 정책이 단순하고 분기가 우리 고유(주문 재시도 금지)라 직접 구현이 "프레임워크 미사용·투명성" 원칙과 일관되며, 재시도 횟수·지연·최종 결과를 그대로 `llm_calls`에 적재한다.
- **회피**: LangChain·LangGraph·LlamaIndex 등 LLM 오케스트레이션 프레임워크 — 흐름의 단순함·투명성·추적 가능성·디버깅·의존성 안정성 다섯 항목 모두에서 비용이 이득을 넘는다.

### 9.3 브로커·시세 — KIS Developers API + `httpx`

- **한국투자증권 KIS Developers API** — 한국 개인에게 *모의·실전·시세*를 단일 공식 API로 제공하는 유일한 선택지. 백테스트→모의→실전이 동일 코드 경로로 굴러간다(설계 불변식의 기반). 시세에 더해 **수급·공매도·대차·프로그램매매**까지 KRX 1차 데이터를 받아 Tier0를 채우고(yfinance·무료 크롤링으로는 얻기 어려운 신호), **잔고·보유·미체결·체결 조회**는 선행 게이트(SQLite vs 실잔고 정합성 대조)의 토대가 된다. **진입 비용 0원** — 증권계좌 1개(입금 0원 OK) + App Key/Secret 무료 발급 + 모의 부가 신청 무료. **모의↔실전 차이는 *엔드포인트 도메인 + 계좌번호* 두 개 설정값뿐**.
- `**httpx`** — KIS REST 호출(동기 사이클) + 실시간 시세·체결 WebSocket(비동기, 실거래 단계) 단일 라이브러리로 충당. 동기/비동기 양쪽을 지원해 사이클 코드와 실시간 코드가 같은 클라이언트를 재사용한다. `requests`는 동기만 지원하므로 차선.
- `**aiolimiter**` — KIS 호출 속도 제한(Token Bucket). 모든 KIS 호출을 *초당 한도의 80%*(안전 마진)로 묶어 rate limit 초과를 막는다(16.3, 한도는 `config/rate_limits.toml`로 외부화·모드별 자동 설정). 의존성을 더 줄이려면 `asyncio.Semaphore`로 대체 가능하나, *시간창 기반* 토큰버킷은 `aiolimiter`가 더 정확하다.
- `**pykrx*`* (선택) — KIS로 받기 어려운 종목 마스터·섹터 분류 보완용. 도입 여부는 KIS 자체 커버리지 확인 후 결정. (KIS 장애 시 KRX 공개 데이터 3차 백업 경로로도 등장 — 16.18·4.4 백업 출처.)
- **회피**: KIS 비공식 커뮤니티 SDK(`python-kis`·`mojito2` 등) — 개인 메인테이너 의존, 메이저 변경 시 매매 코드가 흔들리는 위험. REST 호출이 단순하므로 thin wrapper를 직접 두는 편이 의존성 안정성과 일관된다.

### 9.4 글로벌·매크로 데이터 — `yfinance` · `fredapi`

- `**yfinance`** — S&P500·Nasdaq·SOX·VIX·USD/KRW·DXY 등 Tier1 글로벌 지수·환율. 무료·표준 소스. 빈도 제한·간헐 결측이 있어 데이터 신선도 점검으로 흡수.
- `**fredapi**` — 美10Y금리·美기준금리 등 FRED 공식 매크로 시리즈. *1차 매크로 소스로는* `pandas-datareader`보다 유지보수 빈도가 높아 `fredapi` 우선. 국내장의 미국·세계 동조 분석(삼성·하이닉스 SOX 동조, 환율↔수출주·외국인 수급)의 데이터 기반이다.
- **백업 출처 — `pandas-datareader`(Stooq)**: yfinance는 단일 출처라 차단·장애 시 *매 사이클 핵심인 글로벌 시세*가 끊긴다. 그래서 출처별 폴백을 데이터 어댑터(`data/sources/`*)에 사전 등록한다(16.18) — yfinance→**Stooq**(`pandas-datareader` 어댑터)→Investing.com. 어댑터는 *동일 스키마*를 반환하고 *사용 출처 라벨*을 `data_vintage`에 남겨 어느 사이클이 백업으로 돌았는지 사후 추적한다. (위 fredapi 항의 "`pandas-datareader`는 차선"은 *1차 소스* 비교일 뿐, 백업 어댑터로는 채택한다 — 모순 아님.)
  - **FRED는 별도 백업 출처를 두지 않는다**: 매크로(Tier1b)는 결측 시 매매를 *멈추는 게 아니라* 4.0 원칙대로 *해당 사이클 매크로 신호 비중을 낮춘다*(없는 데이터로 환각을 만드는 것보다 안전). 美10Y는 일별 시장값이라 한두 사이클 결측 영향이 작고, 한은 기준금리는 거의 상수(이벤트성)라 마지막 값으로 버틴다. 당초 백업으로 검토했던 한국은행 ECOS API는 *FRED의 핵심인 美 금리를 국내 금리로 대체하는 것 자체가 논리적으로 맞지 않고*, 추가 키 발급·어댑터 유지보수 비용이 백업 이득을 넘어 도입하지 않는다(불필요한 외부 의존·키를 늘리지 않는다는 원칙).
  - **시리즈별 백업 가용성 검증 (정직 고지 — "美→韓 연결" 핵심 가설 보호)**: 위 폴백은 *지수·금리*에선 검증됐으나, 이 시스템이 의존하는 **VIX·DXY·`^SOX`·원자재 선물(`CL=F`·`HG=F` 등)·개별 ADR**은 Stooq 커버리지가 다르거나 (BDI·LME 비철 일부는) *백업 자체가 없다*. 이들이 결측이면 시스템이 *멈추는 게 아니라 매크로 신호가 조용히 약해져* 더 위험하다(4.0 "결측 시 비중↓"). 그래서 ⓐ *시리즈별* 백업 가용성 매트릭스를 외부 API 레퍼런스 문서에 작성하고, ⓑ **ADR·美 개별주(`TSM`·`MU` 등)는 *KIS 해외주식 시세 API*를 1차 대체 경로로 우선 검토**한다 — 단일 브로커(KIS)로 받으면 yfinance 의존도가 떨어진다. ⓒ 백업조차 없는 시리즈(BDI·LME 비철)는 4장 글로벌 매핑 표에서 *확보 불가·보류*로 명시해 "확보 가능한 데이터에만 전문가" 원칙과 정합시킨다.

### 9.5 공시 데이터 — DART API + (`OpenDartReader` 또는 `httpx` 직호출)

DART 전자공시는 한국 공식·정형 공시로, 일반 뉴스 크롤링보다 노이즈·저작권 위험이 낮아 Tier3 도입 1순위. **DART는 이 계획서 안에서 성격이 다른 두 용도로 쓰이고, 그래서 도입 시점도 다르다.**

- **① 트리거 판정용 조회 (결정론 코드, LLM 미관여) — 이벤트 트리거 사이클과 함께(구현 단계) 도입**: 3.1절 *DART 공시 트리거*가 "보유 종목에 재료성 공시가 떴나"를 판정하는 용도다. DART Open API의 *목록/RSS 조회*로 공시 접수번호·유형코드만 받아, 화이트리스트로 재료성 여부를 *코드가* 판정한다(공시 본문을 LLM에 넣지 않는다). 이건 이벤트 트리거 사이클(3.1절)의 부품이므로 그 폴링 감시 코드를 만드는 *구현 단계에 함께 들어간다. 가벼운 목록 조회뿐이라 `httpx` 직호출로 충분하다.
- **② 공시 본문 분석용 (Tier3a, LLM이 본문을 읽음)**: *공시 본문 해석 전용 분석가*(현재 미편성)가 공시 *본문*을 읽고 해석하는 용도다. 다양한 보고서 파싱이 필요해 `OpenDartReader`(오픈소스 래퍼)를 채택하며, 데이터원·저작권 정리 후 별도 분석가로 도입한다. 도입 전까지 *공시 본문 분석 전문가 에이전트는 만들지 않고 `OpenDartReader` 의존성도 추가하지 않는다*(데이터 없는 전문가는 환각 생성기). 단, 이 보류는 ②에만 적용되고 ①(트리거 판정)과는 무관하다 — ①은 전문가 없이 코드만으로 더 일찍 돈다. (촉매·펀더멘털 분석가가 보는 뉴스 헤드라인·요약은 본문 해석과 별개이며 이미 편성됨 — 5장 Tier3b.)
- **③ 뉴스 헤드라인 (Tier3b, 편성)**: 공시와 별개 데이터원으로 **네이버 검색 API**를 쓴다 — 종목/시장 뉴스 *헤드라인·요약·링크*만 무료(일 25,000회)로 받아 촉매·펀더멘털 분석가 입력에 넣는다. 본문은 저작권 회색지대라 사용하지 않는다. KIS 비공식 SDK 회피 원칙과 동일하게 `httpx` 직호출 thin wrapper로 구현한다.
  - **종목 매칭 노이즈 처리 (필수 — 오염된 헤드라인 차단)**: 네이버 검색은 *범용 뉴스 검색*이라 종목명만으로 쿼리하면 *동명 비상장사·계열사·무관 기사*가 섞인다. 잘못 매칭된 헤드라인을 LLM이 "이 종목 촉매"로 읽으면 헛다리 매수로 이어진다(2.2 "데이터 없는 전문가는 환각"의 변종 — *데이터가 오염된* 전문가). 그래서 수집 단계에서 ⓐ *종목명 + 종목코드/대표 섹터 키워드 조합 쿼리*, ⓑ *발행시각 윈도우*(사이클 직전 N시간, config) 필터, ⓒ 제목·요약에 종목명이 *주체*로 등장하는지 간단 검사로 동명·계열사 오매칭을 거른다. 매칭 신뢰도가 낮은 헤드라인은 *버리고*, 헤드라인 0건 종목은 *촉매 부재*로 명시(없는 데이터로 환각 생성 방지 — 2.5절 촉매 인지 한계와 일관).

### 9.6 결정 오케스트레이션 — 코드 기반 다단계 파이프라인 (프레임워크 없음)

별도 프레임워크 없이 **평범한 파이썬 함수 + `asyncio`** 로 워크플로우를 직접 짠다. 통제·재현·테스트 가능성이 매매의 핵심이라 LangGraph류 추상화는 비용이 더 크다. 사이클의 입력 컨텍스트·프롬프트·모델·응답·체결이 모두 SQLite에 그대로 남도록 추상화 층을 얇게 유지한다. 진입점은 `run_trading_cycle.py` 한 파일.

### 9.7 학습·기억 저장소 — `sqlite3` (stdlib) + `numpy`

- `**sqlite3**` (표준 라이브러리) — 모든 테이블(`decisions`·`agent_predictions`·`trades`·`outcomes`·`lessons`·`shadow_outcomes`)과 집계 뷰(`calibration`)의 단일 백엔드. 개인용·단일 파일·소규모 스키마에 ORM은 과잉이고, 결정 시점에 들어가는 SQL이 1차 자료이므로 추상화로 가리지 않는다. 매매 코어·회고 에이전트·대시보드의 유일한 공유점.
- `**numpy**` — 보정·공헌도·반사실 통계(수축·Wilson 신뢰구간 등)와 인출 스코어러의 레짐 벡터 거리 계산.
- 구조화 인출 스코어러(레짐·섹터·셋업·최근성·보정 가중)로 운용하며, 임베딩·벡터DB는 사용하지 않는다.
- **연결 통일·동시성** — 모든 연결은 `memory/db.py`의 `connect()`만 거치고 직접 `sqlite3.connect()`는 금지한다(`ruff`로 강제 — 4.12). `connect()`가 WAL 모드·`busy_timeout`·`foreign_keys` 등 PRAGMA를 일괄 적용해 매매 코어(write)·회고(write)·대시보드(read-only) 동시 접근을 안전하게 만든다(16.2).
- **마이그레이션 — `Alembic`(ORM 없이 단독) 또는 순번 SQL 파일**: 학습 스키마는 컬럼·메타(`source`·`model_version` 등) 추가가 잦아 손코딩하면 5년치 데이터가 깨진다. `Alembic`(SQLAlchemy 마이그레이션 도구지만 ORM 없이 단독 사용 가능) 또는 자체 순번 파일(`memory/migrations/0001_init.sql`…)로 *forward+backward 둘 다* 작성하고, 실행 전 자동 백업·실행 후 정합성 체크를 건다(16.11).
- **백업·암호화** — 매 사이클 종료 후 `sqlite3 .backup`(또는 연속 백업이 필요하면 `litestream`)으로 atomic 스냅샷을 떠 3-2-1 백업(로컬 2차 + 클라우드, 클라이언트 측 암호화)을 만든다(16.6). 저장 데이터 암호화는 OS 디스크 암호화(**macOS FileVault**, 우리 규모에 충분)를 기본으로 하고, DB 단위 암호화가 필요하면 `SQLCipher`.
- **회피**: SQLAlchemy·SQLModel 등 ORM — 추상화 이득이 작고 감사 가능성과 충돌. 벡터DB·임베딩(Chroma·FAISS·OpenAI embeddings) — 명시적 범위 제외.

### 9.8 데이터 처리·지표 계산 — `pandas` · `numpy` · `exchange_calendars`

- `**pandas`** — OHLCV 시계열, GROUP BY 집계, 워크포워드 슬라이싱), 자본곡선·KPI 등 표 형태 데이터 전반.
- `**numpy**` — MA·RSI·MACD·ATR·실현변동성·52주 고저·백분위 환산을 직접 계산. 지표 식이 단순해서 numpy 20줄이면 끝나고, 직접 짜야 비용·세금·슬리피지 보정이 같은 자리에 들어간다.
- `**exchange_calendars**` — KRX·NYSE 거래일·휴장·반장 판정("시간 맞추기"의 결정론 기반). 한국 공휴일은 음력·대체공휴일 때문에 손코딩하면 버그가 잦으므로 라이브러리에 위임.
- **회피**: TA-Lib·pandas-ta — C 의존성·추가 학습 비용이 "numpy/pandas 직접 계산" 정책의 이득을 넘는다. zipline·vectorbt 등 백테스트 프레임워크 — "통제·재현 우선"에 따라 직접 구현.

### 9.9 스키마·설정 관리 — `pydantic` v2 + `pydantic-settings` + `tomllib`

- `**pydantic`** v2 — decider JSON 스키마 검증(A "타입·범위·필수필드 위반 시 즉시 거부, 부분 파싱 금지"의 자연스러운 구현체), 전문가 응답 동일 스키마 강제, `entry_thesis` 같은 중첩 구조도 깨끗하게 표현. FastAPI 이관 시 동일 모델을 응답 스키마로 재사용한다.
- `**json-repair**` — LLM JSON 출력이 부분 손상(트리밍·escape 오류·콤마 누락)됐을 때의 *2차 폴백*. 검증 파이프라인은 1차 마크다운 코드블록 추출 → 2차 `json-repair`(콤마·따옴표 등 *최소 수정*만, **값 추정 금지**) → 3차 `temperature=0.1` 재호출 순으로 시도한 뒤 pydantic으로 최종 검증하고, 그래도 실패하면 그 호출을 실패 처리한다(부분 파싱을 유효 응답으로 채택하지 않음 — 17.3).
- `**pydantic-settings` + `tomllib*`* — 설정의 단일 로딩 진입점. 설정을 성격에 따라 두 갈래로 나눈다: **① 시크릿**(KIS App Key/Secret·FRED·DART·healthchecks 토큰 등)은 `.env`/환경변수로 — 운영(NCP 서버)에서는 리포에 올리지 않은 서버 로컬 `.env`(권한 `chmod 600`) 또는 OS 환경변수로만 주입(16.5). **② 운영 파라미터**(거래세율·LLM 단가·rate limit·모델 ID·Anthropic 한도)는 날짜별/모델별 `config/*.toml`(`tax_rates`·`llm_pricing`·`rate_limits`·`models`·`anthropic_limits`)로 — 값이 자주 바뀌고 *그 시점 값으로 과거를 재현*해야 하므로 코드·시크릿과 수명을 분리한다. 둘 다 pydantic-settings 2.x의 소스 합성(`TomlConfigSettingsSource` + env)으로 *한 모델*에서 타입 검증과 함께 읽으며, TOML 파싱은 표준 라이브러리 `**tomllib`**(Python 3.11+)가 맡아 외부 의존성이 없다. "파일 경로·시크릿·시간대를 코드에 박지 않고 외부화" 원칙의 구현.

### 9.10 스케줄·운영·로깅

- **외부 스케줄러** — `run_trading_cycle.py` 단일 진입점을 cron 등 *외부에서* 트리거한다. 호스트 비의존 원칙에 따라 스케줄러 코드를 매매 코어에 박지 않는다. **현 단계 채택 = NCP(네이버 클라우드) 서울 상시 서버의 cron**(Phase 0 확정). 당초 1순위였던 GitHub Actions(무료)는 Phase 0 선검증에서 *KIS가 해외 IP를 차단*(러너=Azure, 토큰 발급 ConnectTimeout)함을 확인해 탈락했고, NCP 서울 서버에서는 실전·모의 토큰 발급이 모두 통과(HTTP 200)해 한국 IP 상시 서버로 확정했다. 상시 한국 IP 서버라 GitHub Actions의 제약(정시성 없음·콜드스타트 분 예산·일회성 SQLite 외부 영속화·60일 무커밋 비활성)이 모두 사라진다 — cron 정시 실행, 라이브 SQLite 직접 운영, KIS 호출 안정. 키는 서버의 `.env`(시크릿 저장소)에 두고 `pydantic-settings`가 읽는다. 단 시각 자체는 여전히 *분 단위 정밀 의존 금지*(10:00·14:30은 애초에 여유를 둔 시각). 상시 호스트라 KIS **WebSocket** 실시간 폴링으로 확장할 여지도 생겼다(추후). 더 먼 목표로 집 **맥미니**(가정용 IP·완전 자가소유)도 후보지만, 호스트 비의존 설계라 전환 비용은 작다.
- `**structlog`** (또는 stdlib `logging` + JSON 포매터) — 사이클 단계별 구조화 로그. 결정·체결 행 단위로 JSON 로그를 남기면 SQLite 외에도 외부 로그 수집기(파일·CloudWatch 등)에서 사후 분석이 가능. 감사 트레일을 SQLite 외 채널로도 이중화한다. **회전·보존·마스킹**: 무한 누적이 디스크 풀로 사이클을 죽이므로 일별 회전(`logging.handlers.TimedRotatingFileHandler`, 또는 `loguru`)으로 최근 30일 raw·1년 압축 보존(16.10)하고, App Key/Secret·계좌번호·잔고는 structlog processor로 자동 마스킹한다(16.5).
- **외부 모니터링 (dead-man's switch)** — 사이클 정상 종료/SafeStop 시 외부 모니터링 서비스(**healthchecks.io · Better Stack · UptimeRobot** 중 하나, 14장 결정 항목)로 HTTPS ping을 보내, *정기 사이클 시각이 지났는데 ping이 없으면* 외부에서 알람이 울리게 한다(16.4). 파이썬 라이브러리가 아니라 운영 의존성이지만 1인 무인 운영의 silent failure 차단에 핵심이라 스택에 명시한다. 알림은 Discord 웹훅으로 보낸다.

### 9.11 대시보드 — 현재 Streamlit, 향후 Next.js + FastAPI (가역적)

"매매와 분리된 *읽기 전용* 프로세스, 같은 SQLite로만 결합" 원칙 덕분에 대시보드 영역만 통째 교체해도 매매 코어는 한 줄도 바뀌지 않는다.

**현재(Phase 8) — 1인·내부용 최단 경로:**

- `**streamlit`** — 수익·포지션·매매내역·결정근거·교훈·KPI 시각화. 동일 SQLite를 *읽기 전용 연결*로만 읽음.
- `**plotly*`* — 자본곡선·KPI·전문가 보정 차트. Streamlit과 친화적(`st.plotly_chart`).
- **접속 — NCP 서버에서 돌리고 본인만**: 서버엔 화면이 없으므로 대시보드는 NCP 서버에서 실행하고 **SSH 포트포워딩**(또는 **Tailscale** mesh VPN — 내 계정 기기끼리만 잇는 사설망)으로 *내 기기에서만* 연다. 공개 인터넷 노출은 0을 유지한다(외부 노출 0 → 자본시장법 게이트 정합). Tailscale을 쓰면 폰·외부기기 어디서나 접속하면서도 인증을 Tailscale이 대신해 별도 로그인 구현이 불필요하다(8.6).

**향후 이관 시 — 외부망 노출·반응형 UI·다중 디바이스 접근이 필요해질 시점에 검토:**

- `**fastapi`** — `/positions`·`/decisions`·`/lessons`·`/kpi` 등 *읽기 전용* REST 엔드포인트만 노출. **주문 API는 절대 노출하지 않음**(리스크 엔진 우회 차단).
- `**uvicorn*`* — ASGI 서버. 1인 규모면 단독 + 시스템 supervisor(launchd / systemd)로 충분(gunicorn 과잉).
- `**pydantic**` — API 응답 스키마(에서 이미 사용 중이라 자연스러운 재사용). FastAPI의 OpenAPI 스키마 → 프론트 TypeScript 타입 자동 생성 가능 — pydantic 모델이 그대로 프론트 타입이 되는 게 이관의 가장 큰 이득.
- `**sse-starlette**` (선택) — 사이클 진행·체결 실시간 푸시. Streamlit 자동 새로고침을 대체.
- **프론트엔드(파이썬 영역 밖)**: Next.js(App Router) + TypeScript + 차트(Recharts / TradingView Lightweight Charts) + TanStack Query(데이터 패칭·캐싱).

**이관의 본질**: 매매 사이클은 SQLite로만 쓰고 대시보드는 SQLite로만 읽으므로, Streamlit 시절의 SQL 쿼리·집계 함수가 그대로 FastAPI 엔드포인트의 핸들러로 옮겨 붙는다. 추가되는 파이썬 라이브러리는 위 5개뿐이고, 제거되는 라이브러리는 `streamlit`·파이썬 측 `plotly` 둘이다.

### 9.12 테스트·코드 가드레일 — `pytest` 계열 + `ruff`/`mypy`

- `**pytest`** — 리스크 엔진 룰별 테스트, 데이터 보정 회귀 테스트, 백테스트 골든 케이스. 모드 분기 부재를 검증하는 *paper·live 통합 테스트 매트릭스*(같은 골든 케이스를 두 모드에서 돌려 결과 동일성 확인 — 16.9)도 여기.
- `**pytest-asyncio*`* — `AsyncAnthropic` 등 비동기 코드 테스트.
- `**vcrpy**` 또는 `**responses**` — KIS·DART·yfinance HTTP 호출 녹화/재생. 외부 API 의존 테스트의 결정성을 확보하고 요금·레이트리밋을 회피.
- `**ruff` / `mypy` — 설계 불변식의 기계적 강제(가드레일)**: 단순 린팅을 넘어, 글로만 적으면 1년 뒤 깨지는 불변식을 CI에서 PR 차단으로 박는다 — ⓐ naive `datetime.now()` 금지(`datetime.now(UTC)`만 허용, 16.7), ⓑ `if 실전/모의` 모드 분기 금지(설계 불변식 3.3·16.9, `config/`·`broker/kis_client.py`만 화이트리스트), ⓒ 직접 `sqlite3.connect()` 금지(`memory/db.py`의 `connect()`만, 16.2·16.10). 1인 운영의 가장 큰 적이 *알고리즘이 아니라 silent failure*라는 16장 전제의 코드 측 구현체다.

> KIS·DART API 엔드포인트·호출 한도·토큰 만료 정책은 착수 시 각 포털에서 최신 확인 필요.
