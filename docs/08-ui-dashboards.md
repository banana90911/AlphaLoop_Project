## 8. 플랫폼 (리포트 & 대시보드)

매매 사이클과 분리된 **읽기 전용 프로세스**. 매매 코어는 SQLite에 *쓰기*만 하고, 대시보드는 같은 SQLite를 *읽기*만 한다 — 둘의 유일한 접점은 디스크에 떨어진 SQLite 파일이며, 서로의 가동 여부에 의존하지 않는다. 그래서 대시보드를 통째로 교체·이전해도 매매 코어는 한 줄도 바뀌지 않는다.

### 8.1 대시보드 불변식

요구사항을 구현하기 전에, 대시보드가 항상 지켜야 할 경계부터 못 박는다. 이 경계가 매매 코어의 안전성과 학습 데이터의 무결성을 대시보드 버그로부터 격리한다.

- **읽기 전용 연결만 사용** — 대시보드는 SQLite를 읽기 전용 연결로만 연다. 대시보드에 어떤 버그가 있어도 학습 데이터(`decisions`·`outcomes`·`lessons` …)를 단 한 바이트도 쓸 수 없다.
- **주문·매매 기능 0** — 화면 어디에도 "사자/팔자" 버튼이 없다. 주문은 오직 리스크 엔진을 통과한 매매 코어만 낸다. 대시보드가 주문 경로를 가지면 리스크 검증을 우회하는 구멍이 되므로, *읽기 전용*은 안전이 아니라 *구조적 불변식*이다.
- **매매 코어와 메모리·상태 공유 없음** — 둘은 같은 파이썬 프로세스가 아니다. 전역변수·캐시·큐를 공유하지 않고, 오직 디스크의 SQLite로만 통신한다. 대시보드가 죽어도 매매는 돌고, 매매가 도는 중에도 대시보드는 읽는다.
- **외부 공개 노출 금지** — 결정·수익률·체결·교훈을 공개 인터넷에 노출하면 무인가 투자자문업 위험이 생긴다. 이 제약이 배포·접속 모델을 직접 결정한다.

### 8.2 "최신 상태"의 정의

요구사항의 "실시간으로 파악"을 글자 그대로 *틱 단위 스트리밍*으로 해석하면 과한 설계가 된다. **데이터가 바뀌는 속도가 곧 실시간의 상한**이기 때문이다.

- 매매 사이클은 **하루 정기 2회(10:00·14:30) + 이벤트 트리거**에만 SQLite에 쓴다. 즉 *초 단위로 흐르는 데이터가 애초에 없다*. 그래서 이 시스템에서 실시간의 현실적 정의는 **"최신 사이클 스냅샷 + 자동 새로고침"**이다.
- 대시보드는 N초마다(기본 60초, 설정값) SQLite를 다시 읽어 화면을 갱신한다. 데이터가 하루 몇 번만 바뀌므로 대부분의 새로고침은 같은 화면을 다시 그리지만, 새 사이클이 기록되는 즉시 다음 새로고침에 반영된다. 화면 상단에 **마지막 사이클 시각·데이터 신선도**를 항상 표시해, "지금 보는 게 언제 기준인지"를 명확히 한다.
- **push 실시간(WebSocket·SSE)은 채택하지 않는다** — 그건 *틱 단위로 흐르는 현재가*를 보여줄 때나 의미가 있는데, 그 기능(보유 종목 현재가 실시간 평가)은 상시 호스트 + KIS WebSocket이 필요한 맥미니 단계로 보류한다. 폴링 기반 자동 새로고침이 현 데이터 변화 속도에는 차고 넘친다.

### 8.3 기술스택 — 현 Streamlit + plotly

"현재 Streamlit, 향후 Next.js+FastAPI(가역적)" 결정을 그대로 따른다. 요구사항의 화면 기능이 Streamlit으로 전부 커버되는지를 항목별로 확인한 결과는 다음과 같다 — *현 단계에 Next.js+FastAPI는 과투자*.


| 요구사항 기능                | Streamlit + plotly 구현 수단                 |
| ---------------------- | ---------------------------------------- |
| 시간순 이익 그래프 + 포인트 호버 툴팁 | plotly 그래프의 호버 기능으로 포인트마다 날짜·거래종목·수익을 띄움 |
| 거래 상세 리스트 + 날짜 선택 필터   | 날짜 필터 위젯으로 거른 뒤 표로 표시, 행 선택 시 결정 근거 드릴다운 |
| 클릭형 오류 리포트(클릭→상세)      | 표의 행 선택 또는 펼침 패널로 상세 표시                  |
| 자동 새로고침(최신 상태)         | Streamlit 자동 새로고침 기능                     |
| 주/월별 LLM 비용 추적         | 주·월 집계 + plotly 막대/선 그래프                 |


- `**streamlit`** — 화면 조립. 같은 SQLite를 읽기 전용으로 연다.
- `**plotly**` — 자본곡선·KPI·비용·보정 차트. Streamlit과 친화적이고 호버 툴팁이 요구사항(포인트 호버)을 그대로 만족.
- `**pandas**` — SQL 결과를 표·집계로 가공(주/월 롤업, 날짜 필터).
- **회피**: 현 단계 Next.js+FastAPI 프론트/백 분리 — 1인·내부용·읽기 전용에는 구축·운영 비용이 이득을 넘는다. *외부망 노출·반응형 UI·다중 디바이스·틱 단위 실시간*이 실제로 필요해지는 시점(맥미니 + 외부 접근)에 검토한다.

### 8.4 화면 구성 — 요구사항 → 데이터 소스 매핑

다섯 영역을 *어느 테이블·뷰의 어떤 컬럼*으로 그리는지 명시한다.

**① 나의 정보 (상단) — 잔고·보유 종목·평가손익**

- *보유 종목·수량·평단*: `positions`(내부 보유 스냅샷, `status='open'`)에서 직접. `qty`·`avg_price`·`entry_decision_id`.
- *평가손익*: `positions.avg_price` vs *최신 사이클이 본 가격*(가장 최근 `decisions`/`cycles`에 기록된 그 종목의 시점가)로 계산. 틱 단위 현재가가 아니라 *마지막 사이클 기준* 평가다.
- *현금·총 잔고*: 매 사이클 데이터 수집 단계의 KIS 실잔고 정합성 대조 결과(`positions` 동기화 시점의 계좌 스냅샷). 거래정지 보유분은 `frozen_positions`의 동결 시가로 합산해 자본곡선이 흔들리지 않게 한다.
- *주의*: "잔고"는 KIS 라이브 조회가 아니라 *우리가 마지막으로 기억한 상태*다. 신선도는 상단 사이클 시각으로 고지.

**② 수익 대시보드 (중앙) — 시간순 이익 그래프 + 호버 툴팁**

- *자본곡선*: `outcomes`(청산 실현손익)와 `trades`(체결)를 시간순으로 누적. 동일 KPI 산식 재사용 — **순이익(net, 거래세·수수료·슬리피지 + LLM 운영비까지 차감)** 곡선을 주 곡선으로, 세전·세후는 보조 토글로.
- *포인트 호버 툴팁*: plotly 호버에 그 거래의 **날짜·종목·실현수익(`outcomes`)**을 띄운다(요구사항 그대로).
- *벤치마크 오버레이*: 같은 화면에 4종 벤치마크(코스피 매수후보유·단순 모멘텀·현금·**균등가중 워치리스트**)를 겹쳐 그려, 우리 시스템이 *비용 차감 후* 이들을 이기는지 한눈에 본다(특히 균등가중 워치리스트를 못 이기면 LLM 결정 가치가 0).

**③ 리포트 (좌하) — 거래 상세 리스트 + 날짜 필터 + 결정 근거 펼침**

거래를 *한 줄짜리 요약 리스트*로 쭉 보여주되, 한 거래를 고르면 *그 거래를 왜 했는지*가 같은 화면 아래에 통째로 펼쳐지게 한다. "어떤 거래였나"(요약)와 "왜 그렇게 결정했나"(근거)를 한 화면에서 잇는 것이 이 영역의 목적이다.

- *리스트(한 줄 요약)*: `trades`·`decisions`·`positions`를 시간순으로 한 행에 하나씩. 한 행에 **종목명·섹터(`positions.sector`)·매수가(`trades.fill_price`)·매수일(`trades.filled_at`)·손절예약가(`positions.current_stop_price`)·현재 상태(보유/청산)** 정도의 *식별용 핵심*만 둔다. 날짜 필터로 고르면 그 날짜 거래만 본다.
- *펼침(상호작용) — 행 클릭 → 같은 화면 아래 펼침 패널*: 리스트에서 한 거래(=한 결정)를 클릭하면 그 아래에 그 결정의 근거를 펼친다(Streamlit `st.dataframe` 행 선택 → `st.expander`/컨테이너). *별도 상세 페이지로 이동하지 않고 같은 화면에서 펼친다* — 1인·읽기 전용 복기 도구라 가장 단순·견고하고, 뒤로가기·라우팅 관리가 없다. (순수 마우스오버 hover 툴팁은 Streamlit 표의 행 단위에서 한계가 커 채택하지 않는다 — 풍부한 hover는 ②자본곡선의 plotly 포인트에서만 쓴다.)
- *펼침 패널이 보여주는 근거(이 거래를 왜 했나)* — 출처별로 묶으면:
  - **숫자 근거(코드 산출)**: 모멘텀·수급(외국인·기관 순매수)·거래대금 등 스크리닝/지표 점수와 그 시점 시장 레짐. 출처는 `decisions.context_snapshot`(결정 시점에 결정 에이전트로 들어간 입력 컨텍스트 스냅샷). **보존 주의**: `context_snapshot`은 6개월 지나면 NULL로 압축되므로(6장 보존 정책), *이 숫자 원본은 최근 6개월(Hot) 거래에서만 완전*하다. 옛 거래에서도 모멘텀·수급 같은 *대표 숫자*를 계속 보려면 그 핵심 숫자만 압축에서 살려야 한다 — 아래 *보존 결정* 참조.
  - **결정 에이전트의 답변**: `decisions.rationale`(왜)·`entry_thesis`(촉매·무효화선·R:R·비용 빼고 남는 엣지)·`confidence`·`dissent_addressed`(반대 시나리오 검토)·`regime_tag`. 매수·매도·무거래 모두 사유를 펼친다(`no_trade_reason` 포함).
  - **뉴스·촉매 견해**: `agent_predictions`의 `view`·`confidence`·`key_signals`·`key_risks`(린 구성: 뉴스·결정자). "보내고 받은 것이 1차 자료"의 사용자 측 창구다.
- *청산 결과*: 청산된 포지션은 `outcomes`의 실현손익·보유기간·청산사유(`exit_reason`)를 펼침 패널에 함께 표시.
- **보존 결정 (모멘텀·수급 상세를 옛 거래에서도 보려면)**: `context_snapshot` 전체는 용량이 커 6개월 후 NULL 압축이 맞지만(원칙 유지), *대시보드 펼침에서 늘 보고 싶은 대표 숫자*(모멘텀·수급·거래대금 점수 등 결정 핵심 신호 몇 개)는 크기가 작으므로 **압축에서 살아남는 작은 필드로 따로 보존**한다. 영구 보존되는 `decisions.entry_thesis`(JSON)에 이 대표 숫자를 함께 적재하거나, 동급의 작은 보존 컬럼을 두는 방식 — 정확한 적재 위치·필드명은 6장(`06-data-model.md`) 데이터 모델에서 확정한다(본 절은 *대시보드가 무엇을 보여줘야 하는가*의 요구를 고정하고, 그 보존 수단은 6장이 단일 소스).

**④ LLM 사용량 (우하) — 토큰·비용 주/월 추적**

- 소스는 `llm_calls` — 호출당 `input/output/cache` 토큰, `cost_krw`·`cost_usd`(그 시점 단가로 실측), `agent_role`·`cycle_id`·지연.
- *집계*: 주별·월별 비용 추이(막대/선), `agent_role`별 분해(어느 에이전트가 토큰을 제일 먹나), 거래 1건당 LLM 비용, 캐시 적중률.
- *순이익 연결*: 이 비용이 ②의 순이익 자본곡선에서 *기간 단위로* 차감됨을 명시 표기(LLM비는 사이클 고정비라 개별 체결에 귀속하지 않음).
- *경보 연동*: 일일 비용 한도 80% 도달은 heartbeat 경보 대상 — 대시보드에도 게이지로 노출.

**⑤ 오류·이상 알림 (하단) — 리스트 + 클릭 상세·대처**

- 소스: `safe_stop_events`(안전정지 — 잔고 불일치·데이터 오류·서킷브레이커·모델 이상행동·API 오류·시장 마비 등 `cause`, `released_by`), `trigger_events`(이벤트 트리거 발생·반사실 진단), `cycles`의 `failed_agents`(부분 실패), 데이터 출처 폴백 경보.
- *리스트→상세*: 행 클릭 시 원인(`cause`)·발생 시각·관련 종목·해소 여부(`released_by`)를 펼친다. 일부 SafeStop(모델 이상행동·잔고불일치·데이터오류)은 **사람 개입 해제 필수**이므로, 상세에 *무엇을 확인하고 어떻게 해제하는지* 안내를 함께 보여줘 "대처 가능"을 만족한다.
- *주의*: 대시보드는 읽기 전용이라 *해제 버튼을 누르는 곳*이 아니다 — 어떤 조치가 필요한지 *알려주는* 곳이다. 실제 해제·reconcile는 운영자가 매매 코어 측 절차로 수행.

### 8.5 SQLite를 어떻게 읽나 — 호스트 단계별

대시보드는 SQLite만 읽지만, *그 SQLite 파일이 어디 있느냐*는 매매 코어를 어디서 돌리느냐에 따라 다르다. 두 단계를 구분한다.

- **현 단계 (NCP 서울 상시 서버)**: 상시 호스트라 정식 SQLite를 서버 로컬 디스크에서 *라이브로* 운영한다. 매매 cron과 대시보드를 같은 서버에서 돌리면 대시보드는 *그 라이브 SQLite 파일을 직접* 읽는다(WAL 모드라 매매 코어가 쓰는 중에도 동시 읽기 가능). GitHub Actions 일회성 때 필요했던 "스냅샷 외부 영속화 → 내려받기" 동기화 단계가 없어 가장 단순하다(외부 백업은 3-2-1로 별도 수행).
- **더 먼 목표 (맥미니 — 상시 호스트)**: NCP와 동일하다 — 매매 cron과 대시보드를 한 호스트에서 돌리고 라이브 SQLite를 직접 읽는다. NCP→맥미니는 *실행 위치만* 바뀐다.

두 단계 모두 **대시보드 코드는 동일**하다 — DB 경로 설정만 바뀐다(호스트 비의존). 읽기 전용 연결만 사용하는 규칙도 여기서 동일하다.

### 8.6 배포·접속 모델 — 단계적, 호스트 비의존

"웹사이트", "오직 나만 접속", "최신 상태로 파악"과 "외부 공개 노출 금지"를 동시에 만족시키는 길은 *공개 인터넷에 띄우지 않는 것*이다. 어떤 접속 모델이든 "대시보드를 어디서나 보려면 대시보드 호스트가 상시 켜져 있어야 한다"는 제약이 있는데, *이제 NCP 상시 서버가 있으므로* 대시보드를 그 서버에서 돌리고 사설 경로로만 접속한다.

**현 단계 채택 — NCP 서버 Streamlit + 사설 접속 (공개 배포 없음)**

- NCP 서버에서 Streamlit을 띄우고 라이브 SQLite를 직접 읽으며, **SSH 포트포워딩**(또는 **Tailscale** mesh VPN — 내 계정 기기끼리만 잇는 사설망)으로 *내 기기에서만* 연다. 공개 인터넷 노출 0 → 자본시장법 게이트와 완전 정합. "오직 나만 접속"은 사설 경로 인증으로 충족(Tailscale을 쓰면 폰·외부기기 어디서나 접속하면서도 별도 로그인 구현이 불필요).
- *왜 이걸로 충분한가*: 매매는 이미 NCP 서버에서 **무인으로** 돈다(내가 안 봐도 거래한다). 그래서 대시보드는 *실시간 감시*가 아니라 **사후 확인·복기** 도구다 — "어디서나 늘 떠 있어야 한다"는 요건의 실제 강도가 약하다.
- *왜 대시보드를 공개 배포(외부 노출)하지 않나*: 공개로 띄우려면 ⓐ 로그인 인증, ⓑ 자본시장법에 걸리는 *공개 표면*, ⓒ 비용·운영을 떠안아야 한다. 사설 접속(SSH/Tailscale)이면 이 부담 없이 "어디서나"를 얻으므로, 1인·읽기 전용 대시보드엔 공개 배포가 이득보다 비용·법적 표면이 크다.

**더 먼 목표 — 맥미니 확보 시: 동일 모델(가정용 IP)**

- NCP와 동일하게 매매 cron과 대시보드를 한 호스트에서 돌리고 **Tailscale** mesh VPN으로 접속한다. 맥미니의 추가 이점은 *가정용 IP·완전 자가소유*(데이터센터 IP 의존 제거)이며, 공개 인터넷 노출 0은 동일하게 유지된다.
- 이전 비용은 작다 — 호스트 비의존 설계라 대시보드 코드는 그대로고, 실행 위치(NCP→맥미니)만 바뀐다.

### 8.7 단계적 도입 (Phase 매핑)

- **Phase 8 (현)**: NCP 서버 Streamlit + plotly, 읽기 전용. 위 다섯 영역 + 자동 새로고침. 라이브 SQLite 직접 읽기, SSH 터널/Tailscale 접속.
- **맥미니 이전 시**: 같은 대시보드를 맥미니에서 호스팅 + Tailscale 접속 + 라이브 SQLite 직접 읽기. 코드 불변, 설정만 변경.
- **외부망·반응형·틱 실시간이 실제 필요해질 때**: Next.js+FastAPI 이관 + KIS WebSocket 기반 보유 현재가 실시간 평가 + push 갱신. 매매 코어는 그대로(SQLite로만 결합).
