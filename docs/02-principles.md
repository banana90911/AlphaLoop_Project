## 2. 핵심 설계 원칙

### 2.1 LLM은 제안하고, 결정론적 코드가 집행한다

> 모든 판단을 LLM에게 주는 것이 아닌 사전 정의된 기준을 거쳐 LLM의 판단을 검증하고 보강한다.
> LLM 출력은 *제안 주문*일 뿐이다. 코드의 하드 리스크 검증을 통과해야만 실제 주문이 나간다.

지표 계산·손익 계산·리스크 검증·서킷브레이커·주문 집행은 **전부 코드**가 한다. LLM에는 절대 위임하지 않는다. 이것이 "파산 방지 안전장치"의 실체이다.

### 2.2 프로젝트의 핵심은 모델이 아닌 데이터·워크플로우이다

> LLM 외 주가에 영향을 미치는 다양한 데이터를 사용하여 컨텍스트를 확보한다. 
> 데이터를 정제하고 최적의 워크플로우를 설계하여 환각을 줄인다.

모델 추론력은 교체 가능한 상품이다. 이 프로젝트의 가치·해자는 **① 어떤 데이터를 ② 어떻게 구조화·압축해 ③ 어떤 에이전트에 라우팅하고 ④ 결정론적 코드로 어떻게 집행하느냐**에 있다. 
데이터 없는 전문가 에이전트는 환각 생성기이므로, **확보 가능한 데이터에만 전문가를 붙인다.**

**분업 원칙 — 코드는 숫자, LLM은 글**: 위 ②③의 따름결과로, *LLM은 코드가 못 하는 일에만* 쓴다. 코드가 못 하는 일은 단 하나, **글(자연어)을 읽고 의미·맥락을 판단·생성하는 것**이다. 지표·수급·매크로·섹터강도·레짐 같은 *숫자* 신호는 코드가 계산하므로, 그 숫자를 *말로 다시 설명*하는 LLM 분석가는 토큰·환각만 더하고 새 정보가 없다. 그래서 **원래 구상했던 분야별 전문가 에이전트(지표·수급·매크로·섹터·레짐 분석가)는 두지 않고**, LLM은 *글 작업*인 **① 뉴스·공시 글을 읽고 촉매를 판단(뉴스 에이전트) ② 숫자 점수와 글 판단을 섞어 최종 결정(결정 에이전트) ③ 청산된 매매의 결정·결과 기록을 읽고 다음 사이클에 주입할 교훈을 글로 정리(회고 에이전트)** 세 곳에만 둔다. 셋 다 코드가 못 하는 글 작업이라는 점에서 한 원칙의 적용이다 — ①②는 *바깥 글(뉴스)·숫자*를 읽어 앞을 보고, ③은 *우리가 남긴 결정·손익 기록*을 읽어 뒤를 본다. ③에서도 숫자 몫(신뢰도 보정·공헌도 분해·반사실 추적 등)은 코드가 계산하며, 회고 LLM은 그 계산 결과를 *옮겨 쓸 수 있는 교훈 문장*으로 바꾸는 데만 쓴다. ①②는 실거래 사이클 안에서, ③은 청산 뒤 학습 루프에서 돈다. (이 세 자리도 확정이 아니라, 각자 *비용을 떼고도 코드만 한 버전을 이긴다*는 백테스트 증명을 통과해야 존속한다 — 아래 문단.)

**자본 규모가 LLM 사용 범위를 정한다 (정직 고지)**: 이 프로젝트의 실제 투입 자본은 *검증 10~30만원·본격 ~100만원* 규모다. 이 규모에선 LLM *월 고정비*가 수익률을 직접 깎는다 — 풀(분석가 6명) 구성의 월 6-10만원은 100만원 자본의 월 6-10%로, *수익을 내기 전에 AI 사용료로 적자*가 된다. 따라서 에이전트 수는 "많을수록 똑똑"이 아니라 **자본 대비 비용이 수익을 잠식하지 않는 선**에서 정해지며(현 린 구성 = 뉴스·결정이 진입 사이클당 1-2회 + 회고가 청산 건당 1회로 학습 루프에서 별도, 합산 월 1만-1.5만원), 각 LLM은 *비용을 떼고도 코드만 한 버전을 이긴다는 백테스트 증명*을 통과해야만 존속한다. LLM은 *개발·백테스트·공부*에 자유로이 쓰되, *소액 실거래 사이클*에는 검증된 최소만 남긴다.

### 2.3 코드 기반 다단계 파이프라인 (프레임워크 미사용)

> 매매에선 "방금 LLM에 무엇을 보냈고 무엇을 받았나"가 1차 자료이다. 매매 로깅이 가능하도록 투명성을 확보한다.

LangChain·LangGraph·LlamaIndex 같은 LLM 오케스트레이션 프레임워크를 사용하지 않는다. 사이클의 각 단계는 평범한 파이썬 함수로 짜고, LLM이 끼는 칸은 Anthropic SDK를 직접 호출한다. 병렬화는 표준 `asyncio.gather`로 처리한다. 즉, 전 워크플로우(오케스트레이션, 프롬프트 조립, 메모리, 검색/인출 - 구조화 스코어링 등)는 **프레임워크를 사용하지 않고 직접 구현한다**. 그 이유는 다음과 같다.

- **흐름이 단순하다**. 프레임워크를 사용하면 코드 줄 수는 줄 수 있으나, 프레임워크를 익히고 추상화를 우회하는 비용이 그보다 크다.
- **투명성**이 보장된다. 매매에선 보내고 받는 내용이 사후 분석의 1차 자료이다. 프레임워크는 시스템 프롬프트·메시지 포맷팅·응답 파싱·도구 루프를 자체 추상화 안에서 처리하므로, 이 1차 자료가 객체 내부에 묻혀 즉시 확인하기 어렵다. 직접 호출이면 페이로드·응답을 같은 화면에서 본다.
- **추적 가능성**: 결정 한 건마다 입력 컨텍스트·프롬프트·모델·응답·체결을 그대로 SQLite에 남긴다. 자가 감사용이자, 실전 전환 시 규제(예: 주문 결정의 audit
trail) 친화. 추상화 층이 두꺼울수록 "저장 가능한 형태로 다시 꺼내는" 부수 비용이 든다. 
- **디버깅**: 이상한 JSON·비용 폭증·잘못된 제안 주문이 발생했을 때 스택 트레이스가 우리 코드 안에서 끝나야 빠르다. 프레임워크 내부에서 끝나면 내부 동작을 학습해야 원인을 짚는다.
- **의존성 안정성**: LangChain류는 메이저·마이너 사이에서도 표준 사용법이 바뀐 전력이 있다(예: LLMChain → LCEL | 파이프 전환). 매매 코드는 잘 도는 동안 안 건드리는 게 미덕이라, 외부발 비호환 변경 압력을 줄인다.

왜 Anthropic SDK가 적합한가에 대한 답은 아래와 같다.

- **단일 공급자**: 분석·결정·회고 모두 Claude 단일 공급자로 가는 상황에선, 여러 LLM을 공통 인터페이스로 묶는 멀티공급자 프레임워크의 핵심 가치가 사라진다.
- **API 1:1**: `client.messages.create(...)` 한 호출에 보낸 파라미터·받은 응답이 1:1로 매핑된다. 그 페이로드를 그대로 로깅하면 재현·감사가 끝난다.
- **필요한 모든 기능이 표면에 노출**: 프롬프트 캐싱(시스템 프롬프트·도구정의 캐시로 비용 절감), 구조화 JSON 출력, 도구 사용, 비동기(AsyncAnthropic)가 추가 래퍼 없이 SDK에 그대로 있다.
- **asyncio와 자연스러운 결합**: 여러 LLM 호출(이벤트 동시 처리·향후 에이전트 추가 등)을 묶어야 할 때 `asyncio.gather(...)` 한 줄이면 별도 그래프·실행기 라이브러리 없이 필요한 병렬도가 표현된다. (린 구성에선 사이클 내 호출이 *뉴스→결정* 1~2개뿐이고 본래 순차라 병렬 필요성 자체가 작다 — 3-2장.)

### 2.4 학습 = 폐루프 메모리 학습

> 시스템은 같은 실수를 반복하지 않고, 과거 데이터를 활용하여 더 나은 선택을 한다.

Fine-tuning과 같은 LLM 모델 자체를 학습시키는 방법도 있으나 해당 프로젝트에서는 이처럼 가중치를 학습하지 않는다. 현실적·효과적 학습은 **결정 → 실제손익 → 교훈 → 다음 결정 컨텍스트 주입**의 in-context 피드백 루프이다. 모든 결정+근거를 기록하고, 청산 시 손익을 기록하고, 회고 에이전트가 교훈을 작성하고, 다음 결정 시 관련 교훈·유사 과거 상황을 검색해 컨텍스트로 주입한다. (강화학습·가격예측 모델 등 진짜 ML은 핵심 밖, 추후 옵션.)

**인출(retrieval)은 학습과 동급의 1급 설계 대상이다.** 여기서 "인출"은 결정 직전 저널에서 *지금 상황에 유효한 교훈만* 골라 주입하는 것을 뜻한다. 노트를 잘 채워도(학습) 맞는 페이지를 못 펴면(인출 실패) 학습은 무쓸모다. 인출 방식은 **구조화 스코어링으로 확정**한다: 레짐 유사도 + 섹터/종목 + 셋업 유형 + 최근성 + 전문가 보정 가중. **임베딩/RAG는 의도적 범위 제외** — 기억은 자가생성 *정형* 데이터라 구조화 매칭이 더 정밀하고, 소표본에서 의미검색은 "말만 닮은" 교훈을 확신하며 끌어오는 오인출 위험이 크다(잘못된 교훈을 확신하며 인출하는 것은 아무것도 안 하느니만 못하다). 구조화 인출이 *측정상* 불충분함이 입증될 때만 재검토하며, 현재 계획된 작업은 없다.

### 2.5 현실적 기대치

> 더 나은 예측을 위해 그에 상응하는 데이터를 사용한다.

LLM 재량매매가 시장을 이긴다는 검증된 근거는 없으며 실제 손실 위험이 있다. 이 프로젝트의 가치는 **엔지니어링·학습 + 체계적 규율·리스크 통제 자동화**에 둔다. 그래서 모의투자 우선이 정답이고, 결정론적 리스크 레이어 + 소액이 필수다. 수익을 과대약속하지 않는다. 다만 미국·수급·이벤트를 규율 있게 통합한다는 점에서 단순 차트봇보다 구조적으로 우월하다.

- **촉매 인지 한계**: "왜 하필 지금 오르는가"의 이유(촉매)를 시스템이 약하게밖에 보지 못한다. 촉매가 되는 *글*(뉴스·공시)은 뉴스·촉매 에이전트가 함께 읽는다(2.2 ①, 별도 에이전트 아님). 다만 뉴스 헤드라인(Tier3b)은 가볍게 바로 넣는 반면, *공시 본문(Tier3a)은 장문이라* 어떤 유형·어느 섹션만 압축해 넣을지를 정하는 전처리가 필요해 **헤드라인보다 늦게 붙는다 — 법적 장벽이 아니라 압축·노이즈 처리라는 엔지니어링 순서 때문**이다. 공시 본문이 붙기 전까지는 촉매·재료를 헤드라인 수준에서만 보아 깊이가 얕으므로, 매수 판단은 이 약점을 전제로 보수적으로 하고 촉매가 불명확하면 확신도를 낮춘다.

### 2.6 수익 측정 · 정량적 학습 · 사이징·청산이 방향만큼 수익을 좌우한다

> 학습 내용을 기록하고, 수익을 측정한다.

- **수익은 측정 가능해야 한다**: "AI가 똑똑해 보인다"는 수익의 증거가 아니다. 코스피 매수후보유·이평 돌파 같은 공짜로 따라할 수 있는 단순 전략들과 비교해 거래비용까지 빼고도 이겨야 비로소 "이 시스템이 가치가 있다"고 말할 수 있다. 이를 이기지 못하면 API 비용·세금·실수 위험만 더한 비싼 차트봇과 다름 없다.
- **학습은 정량적이어야 한다**: 일기 쓰듯 "다음엔 조심하자"는 학습이 아니라, 숫자로 따져 다음 행동을 바꾸는 것이 학습이다. 2.4의 폐루프를 ① 신뢰도 보정(전문가가 80% 확신했는데 실제로는 50%만 맞으면 그 확신을 깎아 듣기) ② 공헌도 분해(손익을 어느 전문가·신호 탓인지 귀속) ③ 반사실 추적(거부·미진입한 매매의 "했다면" 결과로 리스크룰이 과한지 확인) ④ 레짐 정합 검색(과거 교훈을 오늘과 비슷한 시장 국면 것으로 선별)으로 닫는다. 하루 1~2건뿐이라 한두 건짜리 교훈은 운일 수 있어, 최소 표본수를 넘겨야 신뢰한다(운과 실력을 통계로 구분).
- **사이징·청산이 방향만큼 수익을 좌우한다**: 종목을 맞히는 것(방향)은 절반일 뿐, 얼마를 사느냐(사이징)와 언제·어떻게 파느냐(청산)가 실제 수익을 그만큼 좌우한다(같은 종목을 맞혀도 베팅 크기와 청산이 결과를 가른다). 스윙 수익은 "이익은 길게·손실은 짧게"에서 나오므로, decider는 방향뿐 아니라 리스크 예산(사이징)과 청산 계획을 함께 내고, 보유 포지션은 매 사이클 "산 이유가 아직 유효한가"를 재평가한다(사놓고 방치 금지).
