## 12. 단계별 구현 로드맵 (생성 예정 파일)
- **Phase 0 — 스캐폴딩**: 리포 구조, `config`(모의/실전 플래그·리스크 파라미터), 비밀키 관리, SQLite 스키마 생성 (9장 테이블·뷰 단일 소스). **운영 신뢰성 1차 박기 (16장 16.16 우선순위)** — 16.1 사이클 idempotency(`cycles` 테이블·상태머신·`recover_pending_cycles()`), 16.2 SQLite WAL/busy_timeout/synchronous PRAGMA 정책 + `memory/db.py` 단일 진입점, 16.5 시크릿 저장소 결정·환경변수만 사용, 16.7 시간대 정규화 경계 함수·CI 룰, 16.9 "if 실전 분기 금지" `ruff` 룰. **첫 배포 환경(16.12) 확정**(맥미니 cron 또는 클라우드)과 **시크릿 저장소·백업 도구·알림 채널**(14장 결정 항목) 동시 결정. **17장 LLM 운영 정책 스캐폴드** — 17.1 재시도·백오프·17.4 프롬프트 캐싱 `cache_control` 경계 정의(아직 LLM 호출은 없지만 *호출 어댑터 정책*은 박아둠)
- **Phase 1 — KIS 연동(모의)**: 인증/토큰, 잔고·시세 조회, 모의 주문 — `broker/kis_client.py`
- **Phase 2 — 데이터 레이어**: 후보 선별(스크리닝) — `data/screener.py`(워치리스트 산출); Tier0 시세·지표·수급 적재 — `data/market_data.py`; Tier1 글로벌·매크로 — `data/global_macro.py`; Tier2 이벤트 캘린더 — `data/calendar.py`
- **Phase 3 — 리스크 엔진**: 하드룰·서킷브레이커 — `risk/risk_engine.py` (결정 파이프라인보다 먼저 머지). 사이징·청산·포트폴리오 구현(`risk/sizing.py`·`exec/exits.py`·`risk/portfolio.py`)
- **Phase 4 — 결정 파이프라인**: 전문가 6 + 결정 에이전트 프롬프트·구조화 출력 — `agents/`(`technical.py`·`flow.py`·`macro.py`·`event.py`·`portfolio.py`·`devils_advocate.py`·`decider.py`), `pipeline/trading_cycle.py`
- **Phase 5 — 기억·학습 루프 (정량)**: 저널 기록·SQL 검색, 회고 에이전트 — `memory/journal.py`, `agents/reflection.py`. 교훈 텍스트에 더해 ① 전문가별 신뢰도 보정 저장·집계(`calibration`) ② 반사실(거부·미진입) 섀도 결과 추적(`shadow_outcomes`) ③ **인출 스코어러**(relevant 정의=레짐·섹터·셋업·촉매 정합 / 가중 스코어=레짐유사도·섹터·종목·셋업·최근성·보정 / 주입 상한=교훈 개수·토큰 / 모순 교훈 충돌 해소) ④ 소표본 과적합 가드(최소 표본수·교훈 신뢰도·만료·중복제거·"이 교훈이 실제 도움됐나" 메타추적). 모듈·SQL·스코어 구현.
- 기억은 자가 생성 정형 데이터이므로 SQL 구조화 인출만 사용. **벡터·임베딩 미사용(범위 제외)** — 고전 RAG(parsing/chunking/embedding/vector index)는 비정형 문서용이라 정형 자가생성 기억엔 부적합. 인출은 구조화 스코어링으로 충분하며, 구조화 recall이 측정상 불충분할 때만 재검토(현재 계획된 작업 없음)
- **Phase 6 — 백테스트·성과평가 하니스**: 과거 데이터 재생(비용·체결현실 반영, 룩어헤드·생존편향 차단), 워크포워드·OOS 분리(과최적화 방지), 벤치마크 대비, KPI 산출, Go/No-Go 게이트 — `backtest/engine.py`, `backtest/walkforward.py`, `eval/metrics.py`, `eval/gate.py` (모의 검증보다 먼저 머지 — 수익 진위 검증의 토대). 엔진·벤치마크·비용·지표·게이트·워크포워드 구현.
- **Phase 7 — 모의투자 *기능* 검증**: 스케줄러 연결, 검증 기간 무인 운영, 9단계 파이프라인이 끝까지 도는지 확인. 합격증의 성격은 *기능 합격증*에 한정된다 — 파이프라인 흐름, 리스크 엔진 룰 동작, LLM 응답 스키마 통과·실패 시 거부 동작, SQLite 기록 무결성, 잔고 정합성 게이트, KIS 호출 패턴·토큰 갱신·rate limit 처리, cron heartbeat, 시간대 정규화, 로그 회전 등. *전략 합격증*(샤프·MDD 같은 수익 KPI의 실거래 적용 가능성)은 KIS 모의의 슬리피지·체결 단순화와 모의 미지원 API(예약주문·실현손익조회·기간손익조회 등 약 74%)로 인해 본 단계에서는 줄 수 없다(상세 근거: 5.2절 "모의로 검증 가능 vs 불가능의 경계"). Phase 7 KPI는 *모드 갭 패널티 할인 후*에만 Go/No-Go 게이트에 입력한다.
- **Phase 7.5 — 소액 실거래 *전략* 검증** (신설): Phase 7 기능 합격 후, *실제 자본의 극히 일부*(예: 전체 자본의 5~10%, 종목당 ADV 1% 룰 통과 가능 범위 내)로 *실거래 모드*에서 1~3개월 운영한다. 검증 대상은 KIS 모의가 줄 수 없는 합격증 — 실제 슬리피지·미체결률·체결 품질, VI/상하한가 락업 시 실동작, 외국인 수급 잠정치의 실신호 강도, 실현 손익 분포·Sharpe·MDD, calibration(전문가 보정)의 실거래 적중률, NXT/SOR 라우팅 동작(해당 종목이 워치리스트에 들 시), KIS 토큰 만료·rate limit·WebSocket 끊김의 실대응, 운영 알림(silent failure 감지)의 실효성, 8장 ORD_DVSN 22 스톱지정가가 *모의에서 동작하지 않더라도 실거래에서는 동작하는지* 직접 확인. 본 단계의 손실 시 자본 영향은 *전체 자본의 단일 종목 손절 1R 수준*으로 설계상 미미하므로 *학습 비용*으로 본다. 본 단계 결과(실거래 KPI, 모드 갭 측정값, 운영 이슈 로그)가 Phase 10 본격 실거래 게이트의 1차 입력이다. 자본의 5~10%라는 비중은 자금 규모 확정 후 *최소 의미 있는 사이즈*(슬리피지 측정이 통계적으로 의미를 가질 만한 거래 횟수가 1~3개월 안에 나오는 수준)로 재조정한다.
- **Phase 8 — 대시보드**: Streamlit (`dashboard/app.py`) — 수익/포지션/매매내역/결정근거/교훈 + KPI·전문가 보정·학습 루프 효과. Phase 7 모의·Phase 7.5 소액 실거래·Phase 10 본격 실거래의 KPI를 *별도 탭*으로 표시해 모드 갭(모의가 실전 대비 얼마나 부풀려지는지)을 시각적으로 확인할 수 있게 한다.
- **Phase 9 — 데이터 확장(옵션)**: Tier3a DART 공시 *본문 분석* 전문가(4.5절 ②). *뉴스 헤드라인(3b)·KIS 무료 펀더멘털(4b, 추정실적·목표주가·투자의견)은 이미 촉매·펀더멘털 분석가로 현 범위에 편성됐고, 커뮤니티 감성(3c)·유료 컨센서스(4c)는 범위 제외/후순위이므로 여기서 새로 도입하는 건 공시 본문을 LLM이 읽는 분석가뿐이다.* DART 공시 트리거 판정용 목록/RSS 조회(4.5절 ①·3.1절)는 이벤트 트리거 사이클의 부품이라 그 폴링 감시 코드 구현 시점에 이미 들어가 있다. DART는 5.2절 데이터 vintage 처리 원칙(수신 시점 스냅샷 동결, 정정공시·취소공시 라벨 분리)을 동일하게 따른다.
- **Phase 10 — 본격 실거래 게이트**: Phase 6 백테스트 OOS 통과 + Phase 7.5 *전략 합격증* + 모드 갭 측정값 반영 후 KPI가 Go 임계 충족 시에만 본격 실거래 자본 투입. *Phase 7 모의 KPI를 본 게이트에 직접 입력하지 않는다*(모드 갭으로 부풀려진 값이기 때문) — Phase 7은 *통과·미통과* 이진 판정만 본다. 본격 실거래 비중도 *한 번에 100% 투입이 아니라* 단계적 자본 램프업(예: Phase 7.5의 10% → 30% → 60% → 100%, 각 단계 1~2개월 운영 후 KPI가 직전 단계 대비 악화되지 않은 경우에만 다음 단계 진입)을 둔다. 어느 단계든 KPI가 직전 단계 대비 *측정상 유의미하게* 악화되면 자본을 직전 비중으로 되돌리고 원인 분석에 들어간다.
## 13. 검증 방법 (End-to-End)
1. **단위**: 리스크 엔진 룰별 테스트 (위반 주문 거부·축소·서킷브레이커·7장 A.1의 9단계 검사 순서)
2. **연동**: KIS 모의계좌로 인증→시세→모의주문→체결기록 1사이클 수동 실행. Phase 1에서 *ORD_DVSN 22 스톱지정가 모의 작동 여부*(8장)와 *모의 vs 실전 체결 갭 측정*(5.2절)도 함께 수행
3. **데이터 검증**: Tier0/1/2 수집·압축 결과가 정확하고 에이전트별 라우팅이 의도대로인지 확인. 5.2절 vintage 정책(yfinance·FRED 스냅샷 동결·KRX 잠정/확정 라벨 분리)이 작동하는지 추적
4. **파이프라인 드라이런**: 실주문 차단 플래그로 전 단계 실행, 전문가 견해·제안 JSON·로그 검증 (악마의 변호인 반대의견이 결정에 반영되는지 + 6장 *베어 우세 시 강제 무거래 룰*이 dissent_addressed < 0.6에서 동작하는지 포함)
5. **운영 신뢰성 검증 (16장)**: `cycles` idempotency(중간 종료 후 재진입 복구), SQLite WAL 동시성, KIS 토큰 만료 자동 갱신, heartbeat 미수신 시 알람, 시간대 정규화 경계, NTP 오차 점검, "if 실전" 분기 금지 CI, 부분 체결 reconcile. `safe_stop_events`·`cycles` 테이블에 기대대로 적재되는지
6. **LLM 운영 검증 (17장)**: 재시도·백오프 동작(429·529·5xx 시나리오 주입), JSON 파싱 실패 4단계 폴백, 프롬프트 캐싱 적중률(`llm_calls` 테이블의 `cache_read_input_tokens` 비율), 부분 실패 시 결정 룰(분석가 N명 미응답 → 무거래 또는 SafeStop)
7. **백테스트·성과 검증(Phase 6)**: 알려진 과거 구간 재생 결과가 *세후* KPI(10장)로 출력되고 Go/No-Go 게이트가 *다중비교 보정*(PBO·Deflated Sharpe) 포함해 작동하는지 확인. *균등가중 워치리스트 매수후보유* 벤치마크와 비교해 결정 가치가 분리되는지
8. **모의 *기능* 검증 (Phase 7)**: 스케줄러로 검증 기간 자동 운영, 대시보드로 손익/규율위반/교훈 축적 모니터링. *기능 합격증*만 발급 — 전략 합격은 9번에서
9. **소액 실거래 *전략* 검증 (Phase 7.5)**: 자본 5~10%로 1~3개월 운영. KIS 모의가 못 주는 합격증(실제 슬리피지·VI 락업·실현 손익 분포·calibration 적중률)을 측정. 결과가 Phase 10 본격 실거래 게이트 1차 입력
10. **정량 학습 루프 확인**: 회고 교훈이 다음 사이클에 주입되는지 + `agent_predictions`·`shadow_outcomes` 적재, 인출 스코어러가 레짐·섹터·셋업 정합 교훈을 우선 반환하고 비관련 교훈을 거르며 주입 상한을 지키는지, 소표본 교훈이 신뢰 임계 미달로 주입 제외되는지 추적. `helped_score`가 *코드*로 산출되는지(6장 메타 환각 차단 정책)
11. **인출품질 측정**: 사후 판정으로 (a) 주입된 교훈이 실제 관련 있었는지 적중률, (b) 교훈 주입 결정 vs 미주입 결정 성과 비교 — "학습한 것을 실제로 잘 활용하는가"를 수치화. 1~2년차에는 9장 *학습 루프 정직 고지*에 따라 칸 채우기 시간 영향으로 측정치가 의미를 갖기까지 시간이 걸린다는 점 고려
